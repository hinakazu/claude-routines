**cronとPATH管理**
• cronジョブはインタラクティブシェル（.zshrc等）の環境変数を引き継がず、HOME/LOGNAME/PATH/SHELLのみが最小限に設定される。`/Applications/cmux.app/Contents/Resources/bin` のような独自パスはcronでは未登録になりexit 127でサイレント失敗する。
• 解決策: crontabファイルの先頭で `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/Applications/cmux.app/Contents/Resources/bin` のように全ジョブ共通のPATHを明示宣言する。
• デバッグ術: cronジョブ内で `env > /tmp/cron-env.txt` を実行し、インタラクティブシェルの `env > /tmp/shell-env.txt` と diff で比較することで欠落した変数を一発特定できる。
• macOS固有のベストプラクティス: cronの代わりにlaunchd（plist形式）を使うと `EnvironmentVariables` キーでPATHを個別指定でき、スリープ復帰後もジョブが確実に実行される。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は一見安全に見えるが罠がある: `call && true` のようにパイプや条件式で繋いだ場合、失敗してもスクリプトが継続してしまうケースがある。サブシェルはデフォルトでERRトラップを継承しないため、関数内エラーが握りつぶされる。
• 堅牢化の3点セット: `set -eE`（errexit + errtrace）+ `set -o pipefail` + `trap 'echo "Error at line $LINENO, exit code $?" >> /var/log/script.log' ERR` を組み合わせることで、どこで何のコマンドが失敗したかをログに残しながら確実に停止できる。
• 意図的に失敗を許容したいコマンドには `command || true` を付けてエラーをスワローし、`set -e` の意図せぬ停止を防ぐ。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーは「自分が最後にpullした後にリモートへ別のコミットが積まれた」状態でpushしようとしたときに発生。自動化スクリプトでCIがコミットを追加するケースでは特に頻発する。
• 正しい解決パターン: `git pull --rebase origin <branch>` → `git push origin <branch>` の順。rebaseは「ローカルのコミットをリモート最新の上にリプレイ」するため履歴が線形に保たれる。mergeでも解決できるが「Merge branch」コミットが増えて履歴が汚れる。
• 自動化スクリプトの鉄則: ファイルコピーや変更は必ず `git pull` の後に行う（`pull → edit/cp → commit → push`）。`git config alias.sync "!git pull --rebase && git push"` のようなエイリアスでpull+pushを1コマンドにまとめるとミスが減る。
