**cronとPATH管理**
• cronジョブはユーザーの.zshrc/.bash_profileを読み込まない最小環境で動作するため、crontabの先頭で `PATH="/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin"` と明示的に設定するのが最も確実な方法。`PATH=$PATH:/追加パス` のような変数展開はcron実装によっては動作しないため避けること。
• macOSでは `launchd` がcronの後継として推奨。`launchd` はスリープ中のジョブスキップがなく、EnvironmentVariables でPATHを明示指定できる。Apple Siliconでは `/opt/homebrew/bin` を必ず含める。また `~` は展開されないため絶対パス `/Users/yourname/` を使用すること。
• cronスクリプトのデバッグには `#!/bin/bash -l`（ログインシェル）を使うか、ラッパースクリプトでフル環境を明示的にセットアップする方式が信頼性・テスト容易性が高い。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体には3つの主要な落とし穴がある：①コマンド置換 `$(false; echo "still runs")` には伝播しない、②算術演算 `((count--))` が0になると終了コード1で予期せず終了、③パイプは前コマンド失敗を無視する。これらを防ぐには `set -euo pipefail` の3オプションを組み合わせることが必須。
• エラー時のログ不足を解消するには `trap 's=$?; echo "$0: Error on line $LINENO: $BASH_COMMAND"; exit $s' ERR` を追加することでエラー行番号と失敗コマンドを記録できる。`set -eE`（大文字E）を追加すると ERR trap がサブシェル・関数・コマンド置換にも継承される。
• これらの実装でデプロイパイプラインのサイレント失敗率が94%低下したという実績がある。スクリプト冒頭に `set -euo pipefail` + `trap` をセットで書くことをルール化すると効果的。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーはリモートブランチがローカルの親コミットでない状態でpushしようとすることで発生。自動化スクリプトでの正しい操作順序は「`git pull --rebase` → ファイル変更 → `git add` → `git commit` → `git push`」。変更後にpullすると unstaged changes が rebase をブロックして失敗する。
• `git pull --rebase` はマージコミットを作らず線形履歴を保つため、CI/CDパイプラインでのデバッグ（git bisect等）が容易になる。自動化スクリプトでは `git pull --rebase origin main` を commit前に必ず実行するか、`git fetch` + `git rebase origin/main` を使うとより明示的に制御できる。
• 複数箇所からpushが発生するリポジトリでは、Mergify等のツールでpull request前のrebaseを自動強制化することでチーム全体の一貫性が保たれ、マージコンフリクトを30〜40%削減できる。force pushが必要な場合は必ず `--force-with-lease` を使用すること。
