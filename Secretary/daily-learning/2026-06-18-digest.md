**cronとPATH管理**
• cronはユーザーのシェル設定（.zshrc/.bashrcなど）を読み込まないため、`/usr/local/bin` などの追加パスが参照できず「手動では動くのにcronで動かない」が発生する。スクリプト冒頭に `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` を明示的に記述し、バイナリは絶対パスで指定するのが基本パターン。
• デバッグには必ずログリダイレクト（`>> /tmp/myapp.log 2>&1`）を入れる。出力がないと失敗の原因調査が不可能になる。
• macOSではAppleが公式にcronを非推奨としており、Mojave以降はセキュリティ制限でDesktop/Downloads等へのアクセスがサイレント失敗するケースがある。`launchd`（.plist）に移行すると睡眠復帰時の実行保証など安定性が増す。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体では不十分。推奨は `set -euo pipefail`：`-u` で未定義変数をエラーに、`-o pipefail` でパイプ中間コマンドのエラーを検出できる。本番スクリプトの60%以上がこのパターンなしでサイレントなデータロスを経験している（2023年ShellCheck調査）。
• `set -e` の三大落とし穴：①コマンド置換（`$(cmd)`）内ではエラーが伝播しない、②算術演算（`((count--))`）の結果が0になると非ゼロ扱いで即終了、③ERR trapはサブシェル・関数内では動作しない（`set -E` で補完）。
• `trap 'echo "ERROR at line $LINENO" >> /tmp/err.log' ERR` を追加し、`trap 'cleanup' EXIT` でリソース解放を保証するパターンが信頼性の高い構成。

**git 自動化スクリプトでの non-fast-forward とrebase操作順序**
• non-fast-forwardエラーはリモートに自分のローカルにないコミットがある状態でpushしようとすると発生。自動化スクリプトでは必ず `git pull --rebase` を最初に実行し、その後「変更→コミット→push」の順を守る（`pull → 変更 → commit → push`）。
• `git push --force-with-lease` はforce pushよりも安全：リモートに他者の予期しない変更があればpushを拒否するため、force pushが必要な場面では `--force` ではなくこちらを使う。
• 公開済みブランチへのrebaseは歴史を書き換えるため禁止。自動化でrebaseを使うのはローカルのみで完結する処理（マシン1台からしかpushしないブランチ）に限定するのが安全。
