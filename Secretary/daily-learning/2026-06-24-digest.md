**cronとPATH管理**
• cronジョブは `.zshrc` / `.bash_profile` を読み込まず、PATH は `/usr/bin:/bin` 程度の最小環境になる。スクリプト冒頭で `export PATH="/opt/homebrew/bin:/usr/local/bin:$PATH"` を明示するのが最も確実な対策。
• crontabファイル内で `PATH=/usr/local/bin:/usr/bin:/bin` と変数設定も可能（`export` 不要）。また `#!/bin/bash -l` でログインシェルとして起動するとプロファイルが読み込まれる。
• macOS では Apple 公式が launchd を推奨。cron はスリープ中のジョブをスキップするが、launchd はスリープ復帰後に遅延実行されるため、モバイルMacでの自動化には launchd が圧倒的に適している。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `if`/`while` の条件部、`&&`/`||` の非末尾、パイプラインの非末尾コマンドでは発動しない。特にコマンド置換 `$(...)` 内の失敗はデフォルトでサイレントにマスクされる（`inherit_errexit` で回避可能）。
• ベストプラクティスは `set -euo pipefail` の3点セット。`-u` は未定義変数をエラーに、`pipefail` はパイプライン途中の失敗をスクリプトに伝播させる。
• `trap 'echo "Error at line $LINENO" >&2; exit 1' ERR` を冒頭に置くだけで失敗した行番号をstderrに出力できる。`-E`（errtrace）オプションで子シェルにもERR trapを継承させる。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでの正しい操作順序は `git pull --rebase` → ファイル操作 → `git add` → `git commit` → `git push`。unstaged changes がある状態で rebase すると `error: cannot rebase: You have unstaged changes` でブロックされる。
• `git config pull.rebase true` を設定しておくと `git pull` が常にrebaseモードで動く。さらに安全を期すなら `git stash` → `git pull --rebase` → `git stash pop` のパターンで未コミット変更を保護する。
• 複数箇所からpushが発生するリポジトリでは、スケジュールで `git fetch` してブランチ乖離を早期検知するコンフリクトチェックスクリプトを用意しておくと、non-fast-forwardに気づくタイミングが早くなる。
