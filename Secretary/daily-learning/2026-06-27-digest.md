**cronとPATH管理**
• cronの実行環境は `~/.zshrc` や `~/.bashrc` を読み込まず、デフォルトPATHは `/usr/bin:/bin` のみ。`/usr/local/bin` や `/opt/homebrew/bin` など追加パスは全て未登録となり、コマンドが exit 127 でサイレント終了する。
• 最も確実な対策はスクリプト冒頭で `export PATH=/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin` を明示するか、コマンドに絶対パスを使うこと（例：`claude` → `/Applications/cmux.app/Contents/Resources/bin/claude`）。
• crontabエントリに `PATH=... ; /path/to/script.sh` と直接書く方法もある。ただし `PATH=$PATH:/追加パス` は `$` が展開されないcron実装があるため避けるべき。推奨ヘッダーパターン：`#!/usr/bin/env bash` + `set -Eeuo pipefail` + 明示的なPATH宣言。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体では「パイプライン途中の失敗」「コマンド置換 `$()` 内のエラー」「`&&`/`||` リストの途中失敗」は検出できず、無音終了する。必ず `set -euo pipefail` の3つセットで使う。
• `trap 'echo "ERROR at line $LINENO: $BASH_COMMAND" >&2' ERR` でどのコマンドが何行目で失敗したかをログに残せる。さらに `set -E`（errtrace）を追加するとサブシェル・関数内にも ERR トラップが継承される。`inherit_errexit` を追加するとコマンド置換にも `errexit` が伝播する。
• クリーンアップ（tmpファイル削除など）は `trap ... EXIT` に書く。`EXIT` トラップは `set -e` による強制終了時にも必ず発火するため、エラー時と正常終了時の両方をカバーできる。`ERR` トラップとの重複を避けるため、ロギングは `ERR`、後片付けは `EXIT` と役割を分担するのが定石。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでの安全な git 操作順序は `git stash → git pull --rebase → git stash pop → ファイル操作 → git add → git commit → git push`。`git pull` 前に未コミット変更があると rebase がブロックされ、それがサイレントに自動化を止める。
• `git rebase --autostash`（または `git config --global rebase.autoStash true`）を使うと stash/pop を自動化でき、操作ミスを防げる。non-fast-forwardエラーはリモートブランチにローカルにないコミットが存在する場合に発生し、Gitがリモートへの上書きを防ぐための安全機構。
• Git 2.34 以降、`git pull` のデフォルトは fast-forward できない場合に abort（`--ff-only` 相当）に変更された。スクリプト内では `git pull --rebase` と明示するか `git config pull.rebase true` を事前設定しておくことが推奨。
