**Claude Code MCPツールスコープとサブエージェント設計**
• サブエージェントはデフォルトで親の会話のツールセットを継承するが、frontmatterの `tools`（ホワイトリスト）と `disallowedTools`（ブラックリスト）で明示的に制限できる。lazyweb のような MCP ツールを特定のサブエージェントにのみ渡すには、`.claude/agents/` のサブエージェント定義にインラインで MCP サーバーを記述するのが最も確実な方法。
• サブエージェントはサブエージェントを生成できない（無限ネストを防ぐ制約）。そのため、メイン Claude が lazyweb で情報収集し、指示だけをサブエージェント（桂小五郎）に渡す設計は正しく、この制約を前提にしたアーキテクチャ。
• 2026年時点で MCP は月間 9700 万 SDK ダウンロードを超えた業界標準となり、Anthropic・OpenAI・Google・Microsoft・AWS が全社でサポート。サブエージェントごとに最小権限の原則（Least Privilege）を適用するスコープ設計がエンタープライズでは必須となっている。

**LIFF 審査中のローカル開発バイパスパターン**
• LINE 公式の `@line/liff-mock` プラグイン（LIFF SDK v2.19.0 以上）で `liff.use(new LiffMockPlugin())` を登録し、`liff.init({ liffId: 'liff-xxxx', mock: true })` とするだけでモックモードを有効化できる。`liff.$mock.set()` で userId やプロフィール情報を自由に差し替えてテスト可能。
• 審査中はエンドポイント URL を変更できないため Cloudflare Tunnel 等は機能しない。`NEXT_PUBLIC_USE_MOCK=true` のような環境変数でクライアント側の LIFF 初期化をスキップし、API 側でも `mock-id-token-for-dev` のような固定トークンで認証を分岐させる二層バイパス設計が実務上最も安全。
• この設計は「本番コードに開発用バイパスを含む」構造になるため、環境変数でのフィーチャーフラグ管理と CI での `NODE_ENV=production` 時の無効化テストを必ずセットで行う必要がある。

**Unsplash 画像 ID 管理と React onError フォールバック設計**
• Unsplash Source（`source.unsplash.com`）は廃止予定で、新規プロジェクトでは Unsplash API 経由の URL（`images.unsplash.com/photo-{id}`）を使うことが必須。特定フォト ID は削除されると永続的に 404 になるため、モック用 ID リストは定期的に検証する運用が必要。
• React の `onError` フォールバック実装は `<img src={url} onError={(e) => { e.currentTarget.src = '/fallback.png'; }} />` が基本。ただし `src` が `null` や `undefined` の場合は `onError` が発火しないため、`src={url ?? fallbackSrc}` で先にデフォルト値を渡してコンポーネントクラッシュを防ぐ必要がある。
• 外部画像 URL への依存リスクを最小化するには、(1) 本番は Unsplash API から取得した URL をバックエンドで保存して CDN キャッシュ、(2) モック環境は自分のホスティングした画像を使う、という二層設計が理想的。
