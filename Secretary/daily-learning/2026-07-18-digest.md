**cronとPATH管理：自動化スクリプトが手動では動くのにcronで動かない罠**
• cronはログインシェルを起動しないため `.zshrc`/`.bash_profile` が読み込まれず、PATHは `/usr/bin:/bin:/usr/sbin:/sbin` の最小構成になる。`claude`や`node`などHomebrewツールは即座にexit 127でサイレント失敗する
• 解決策①：crontab冒頭に `PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin` を絶対パスで宣言（`$PATH`の変数展開は使用不可）。解決策②：ラッパーshスクリプトにPATHを明示し、crontabからそのスクリプトを呼ぶ。ジョブ固有パスはジョブのインライン指定が推奨
• macOS固有の注意：Apple Silicon MacではHomebrewが `/opt/homebrew/bin` に入るがcron/launchdはこれを自動追加しない。launchdのplistではジョブごとに `EnvironmentVariables` で個別PATH設定可能。launchdはMacスリープ中に実行されなかったジョブを起動後に自動実行する点でcronより信頼性が高い

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` の3大落とし穴：①コマンド置換（`$(cmd)`）にはset -eが伝播しない ②`if`/`while`/`&&`/`||`のコンテキスト内では失敗してもexitしない ③パイプ `cmd1 | cmd2` でcmd1が失敗してもcmd2が成功すれば終了コード0を返す
• 推奨パターンは `set -euo pipefail`（3点セット）＋ `trap 'echo "Error at line $LINENO, exit $?" >&2' ERR`。`set -E` を追加するとサブシェルでもtrap ERRが継承される
• クリーンアップを確実に実行するには `trap cleanup EXIT` と組み合わせる。`local var=$(failing_cmd)` のような local付き代入は終了コードが消えるため、`local var; var=$(failing_cmd)` と2行に分けることが必要

**git 自動化スクリプトの操作順序とnon-fast-forward対策**
• non-fast-forward errorはローカルとリモートが diverge した状態（双方に独立コミットあり）で発生。`git pull --rebase` でローカルコミットをリモート最新の上に replay し直し、マージコミットを作らずリニアな履歴を保つ
• 自動化スクリプトの正しい順序は `git pull --rebase → ファイル操作 → git add → git commit → git push`。「ファイル操作してからpull」だと unstaged changes が残った状態でrebaseがブロックされる
• `git config --global pull.rebase true` でデフォルトをrebaseに固定しておくと `git pull` だけでrebase動作になる。チーム共有のgit hooksは `.githooks/` に保存し `git config core.hooksPath .githooks` で有効化するのが現在のベストプラクティス
