**cronとPATH管理**
• cronのデフォルトPATHは `/usr/bin:/bin` のみ。Homebrewや独自インストールツール（例：`/opt/homebrew/bin`）は未登録のため、`command not found`（exit 127）でサイレント終了する。crontab冒頭か各スクリプト先頭で `PATH=/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin` と明示指定するのが鉄則。
• `PATH=$PATH:/new/path` のような変数展開はcrontabの実装によっては展開されず、文字列 `$PATH` がそのまま設定されてしまう危険がある。絶対パスか、スクリプト内で設定するのが安全。
• macOSではcronよりlaunchdが推奨。launchdは `EnvironmentVariables` キーでPATHを明示でき、スリープ後のジョブ再実行にも対応。cronはスリープ中の実行をスキップするが、launchdは `RunAtLoad` 等で補完できる。
• デバッグ：cron内で `env > /tmp/cron-env.txt` を実行し、シェルの `env > /tmp/shell-env.txt` とdiffを取るとPATH差分が一目瞭然。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は論理式（`cmd || echo "failed"`）の中では機能しない。`||` でつながれたコマンドは「意図的なエラー処理」と見なされ、exit 非0でもスクリプトが続行する。
• `((counter++))` のような算術展開でカウンターが0の場合にexit 1を返してスクリプトが終了する（算術評価の「値が0 = 失敗」という仕様による）。
• 推奨構成は `set -euo pipefail`：`-u` は未定義変数をエラー化、`-o pipefail` はパイプ内の中間コマンドのエラーも捕捉する。
• `trap 'echo "Error on line $LINENO: $BASH_COMMAND (exit: $?)" >> /var/log/script.log && exit 1' ERR` をスクリプト先頭に置くと、サイレント終了ではなく行番号とコマンドをログに残せる。
• 特定コマンドのエラーを無視したい場合は `cmd || true` を使う。ただし多用するとエラーが隠れるため `cmd || { echo "Error: cmd failed" >> error.log; true; }` のように最低限のログは残す。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでの正しい順序は `git pull --rebase` → ファイル変更 → `git add` → `git commit` → `git push`。pullより先にファイルを変更・コミットしてからpullするとrebaseがunstaged changesによってブロックされる。
• `git pull --rebase` はリモートのコミットをローカルのベースとし、ローカルのコミットをその上に積み直す（replay）。マージコミットが生まれず `git log` が線形に保たれる。
• non-fast-forwardエラー発生時の解消：`git fetch origin && git rebase origin/<branch>` → コンフリクト解決 → `git push`。`git push --force` は他人のコミットを消す危険があり、自動化スクリプトでは原則禁止。
• 防止策：push前に `git fetch origin && git status` でdivergated状態を検出し、divergedなら自動rebaseを実施するロジックを組み込む。
