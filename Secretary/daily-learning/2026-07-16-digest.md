**cronとPATH管理**
• cronジョブはデフォルトで `PATH=/usr/bin:/bin` のみ設定され、`.zshrc` や `.bashrc` は一切読み込まれない。Homebrewの `/opt/homebrew/bin` やカスタムツールパスは必ずcrontabの先頭か各ジョブにインラインで明示する必要がある（`export` は不要かつ使用不可）。
• macOSではAppleが10年以上前からlaunchdへの移行を推奨。launchdのplistには `<key>EnvironmentVariables</key>` ブロックでジョブごとにPATHを宣言でき、スリープ復帰後の実行漏れ検知やイベントベーストリガーにも対応。cronは「一時的な用途・失敗許容ジョブ」にのみ使うのが2025年以降の推奨。
• Apple SiliconのMacでlaunchdを使う場合は `/opt/homebrew/bin` を明示しないとbrewインストールツールが見つからない。パスは `~` 展開されないため `/Users/yourname/` で絶対パス指定が必須。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `||` や `&&` の論理式内では発動しない、コマンド置換（`$(...)`）はデフォルトでサブシェルが `set -e` を継承しない（Bash 4.4以降は `inherit_errexit` で解決）、`local var=$(failing_cmd)` では `local` の戻り値0がエラーをマスクする、という3大落とし穴がある。
• 2025年の推奨パターンは `set -euo pipefail` + `trap 'echo "Error at line $LINENO: $BASH_COMMAND (exit $?)" >> /var/log/script.log; exit 1' ERR`。`-E` フラグを併用することでERRトラップが関数・サブシェルにも継承される。
• `(( i++ ))` のようなインクリメント式が0を返すと `set -e` でスクリプトが即終了する罠もある。代替として `i=$(( i + 1 ))` を使うか `(( i++ )) || true` で回避。

**git non-fast-forward pushとautomation操作順序**
• 自動化スクリプトでgit操作を組む正しい順序は `git stash → git pull --rebase → git stash pop → [ファイル操作] → git add → git commit → git push`。pull前にunstagedな変更がある状態でrebaseを実行するとブロックされる。
• push拒否（non-fast-forward）が発生した場合、`--force` より `--force-with-lease` が安全。`--force-with-lease` はリモートが自分の最終pull時の状態と同じかを確認してからforce pushし、他者のコミットを上書きしない。
• CI/CDパイプラインでは、スクリプト実行前に `git status --porcelain` で未コミット変更を検出して早期失敗させるか、`git stash -u`（untrackedを含む）で退避してからpullするのが堅牢なパターン。
