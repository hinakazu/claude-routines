**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` や `.bash_profile` を読み込まず、`/usr/bin:/bin` 程度の最小PATHのみで起動するため、HomebrewやカスタムインストールのCLIは見つからずexit 127でサイレント失敗する。
• 根本解決は3択：①crontabファイルの先頭に `PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin` を明示宣言、②スクリプトの1行目を `#!/bin/bash -l` としてログインシェル経由で起動、③バイナリを絶対パスで指定（例: `/opt/homebrew/bin/claude`）。
• macOSではApple推奨の **launchd** への移行が有効な長期策。`EnvironmentVariables` キーにPATHを辞書形式で書けるため管理しやすく、SIP（System Integrity Protection）との相性もよい。Apple Silicon MacはHomebrew用に `/opt/homebrew/bin` を必ず追加。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体では「パイプ内の失敗」や「未定義変数」は検知しない。`set -euo pipefail` の3点セットにすることで：`-e`（コマンド失敗で即終了）、`-u`（未定義変数をエラー）、`-o pipefail`（パイプ内の途中コマンド失敗も検知）が全て有効になる。
• エラー発生箇所を特定するには `trap 's=$?; echo "$0: Error on line $LINENO: $BASH_COMMAND"; exit $s' ERR` を冒頭に置く。これでどのコマンドが何行目で失敗したかをSTDERRに出力できる。
• 注意点：`-e` は `cmd && true` のように `&&` で後続コマンドと連結すると無効化される。また、サブシェルや関数内でERRトラップを継承させるには `set -E`（大文字）も追加が必要。デバッグ時は `set -x` を一時的に加えるとコマンド実行ログが出る。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーはローカルとリモートの両方に独立したコミットが存在する「分岐状態」で発生する。自動化スクリプトでの正しい操作順序は `git pull --rebase` → ファイル操作 → `git add` → `git commit` → `git push`。ファイル操作の**前**にpullすることでunstagedな変更がpull時にコンフリクトするのを防ぐ。
• `git pull --rebase` はマージコミットを作らず、リモートの最新に自分のコミットを積み直すため、自動化スクリプトのログが線形に保たれる。`git pull.rebase true` をgit configに設定しておくとデフォルト化できる。
• pullに先立ちunstagedな変更がある場合は `git stash` → `git pull --rebase` → `git stash pop` の3ステップか、`git status --porcelain` でチェックしてから処理を分岐させる防御コードが有効。共有ブランチへのforce pushやrebaseは履歴を破壊するため禁止。
