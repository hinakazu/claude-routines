**cronとPATH管理**
• cronジョブはユーザーの`.zshrc`/`.bash_profile`を読まないため、デフォルトのPATHは`/usr/bin:/bin`のみ。nvmやpyenv、Homebrew経由でインストールしたコマンドは見つからずexit 127でサイレント終了する。
• 最も確実な対策はcrontabの先頭に`SHELL=/bin/bash`と`PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`を明示宣言すること。crontab内では`$HOME`などの変数展開は効かないためフルパスで書く必要がある。
• デバッグ時はローカルでcron環境を再現できる: `env -i SHELL=/bin/sh PATH=/usr/bin:/bin HOME=$HOME /bin/sh -c '/path/to/script.sh'`を手元で実行すると「cronでだけ失敗する」問題を事前に検出できる。ログはスクリプト側で`>> /var/log/script.log 2>&1`にリダイレクトするのが鉄則。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e`単体には3つの盲点がある: ①パイプ（`cmd1 | cmd2`でcmd1が失敗しても終了しない）②コマンド置換（`$(fail_cmd)`内は`-e`が継承されない）③条件分岐・`&&`/`||`リスト内のコマンド。セットで`set -eEuo pipefail`を使うのがスタートライン。
• ERRトラップで「どのコマンドが、どの行で失敗したか」をログに残せる: `trap '__error_trapper "${LINENO}/${BASH_LINENO}" "$?" "$BASH_COMMAND"' ERR`。`-E`フラグがないとサブシェル・関数内でトラップが継承されないため注意。
• 算術式`((count--))`はカウンタが0のとき終了コード1を返し`set -e`でスクリプトが停止する罠がある。`((count--)) || true`で回避する。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーの根本原因は「ローカルがリモートの最新を取り込む前にコミットした」状態。自動化スクリプトの正しい順序は `git pull --rebase origin <branch>` → ファイル操作 → `git add` → `git commit` → `git push`。途中でファイルをコピーしてからpullすると未ステージの変更でrebaseがブロックされる。
• rebaseはローカルのコミットをリモートの最新コミットの上に積み直す操作で、マージコミットが不要なためCI/CDパイプラインの履歴がすっきりする。rebaseベースのワークフローでマージコンフリクトが30〜40%削減されるという報告がある。
• 自動化スクリプトでは`git config pull.rebase true`をリポジトリに設定しておくと`git pull`が自動的にrebaseになる。ただし共有ブランチ上のコミットにrebaseを使うと他の開発者の履歴が壊れるため、ローカルフィーチャーブランチのみに限定すること。
