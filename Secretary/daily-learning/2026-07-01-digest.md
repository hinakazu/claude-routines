**cronとPATH管理**
• cronジョブは `.bashrc`/`.zshrc` を読み込まず、最小限のPATHしか持たない（例: `/usr/bin:/bin`）。スクリプト冒頭で `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` を明示するか、バイナリをフルパスで呼び出すことが必須。crontabファイルに直接 `PATH=...` を書く方法もあるが、`export` キーワードは不要かつ使用禁止。
• macOSでは cron より launchd（`.plist`ファイルで設定）が推奨される。launchd は `EnvironmentVariables` キーで PATH を明示設定でき、Apple Silicon では `/opt/homebrew/bin` を必ず含める必要がある。cron と異なりスリープ復帰後にもジョブが確実に実行される点が大きな利点。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e`（errexit）は「コマンド失敗時に即終了」するが、`if`条件内・`&&`/`||`チェーン・パイプライン（末尾以外）・コマンド置換内では発動しない致命的な盲点がある。`false | true` が成功扱いになるのが典型例。
• 推奨パターンは `set -euo pipefail` + `shopt -s inherit_errexit` + `trap 'handle_error "$?" "${BASH_COMMAND}"' ERR` の組み合わせ。`-E`（errtrace）でサブシェル内にもERRトラップを継承させ、失敗したコマンド名・終了コード・行番号を必ずログに残す設計にする。
• `set -e` は「安全装置」ではなく「品質向上ツール」として位置づけ、個別の重要コマンドには `cmd || { log_error; exit 1; }` で明示的なエラーハンドリングを追加するのが本番品質のベストプラクティス。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでgit push失敗を防ぐ鉄則は「pull（fetch+rebase）→ ファイル操作 → add → commit → push」の順序を厳守すること。ファイル操作後にpullを挟むと unstaged changes で rebase がブロックされ、non-fast-forward エラーの連鎖が起きる。
• `git pull --rebase` は `git config --global pull.rebase true` でデフォルト動作として設定できる。競合が生じた場合は `git rebase --abort` で安全に巻き戻せる。
• 複数拠点からpushが発生するリポジトリ（CI/自動化スクリプト/手動）では、push前に `git fetch origin && git status` で ahead/behind 情報を確認するステップを必ず挿入するのが2026年のベストプラクティス。
