**cronとPATH管理**
• cron起動時はユーザーの `.zshrc` や `.bash_profile` が読まれず、最小限のPATH（`/usr/bin:/bin` 程度）しか持たないため、手動では動くコマンドがcronでは `exit 127` で無音終了する。スクリプト冒頭で `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` と明示設定するのが基本。
• crontab内で直接 `PATH=...` を宣言できる（`export` キーワードは不要・不可）。ログ出力の `MAILTO` やタイムゾーンの `CRON_TZ` も同様に設定可能。
• macOSではAppleが推奨する **launchd** を使うと、スリープ復帰後も `StartCalendarInterval` 指定のジョブが確実に実行される。cron比較でPATH問題を回避しやすく、plist形式で環境変数を明示指定できる。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e`（errexit）は一見安全に見えるが、コマンド置換 `$(...)` 内のエラーには伝播しない。パイプでも前コマンドの非ゼロ終了をスルーするため `set -o pipefail` と必ず組み合わせること。
• `-E`（errtrace）オプションを加えると ERRトラップが関数やサブシェルにも継承される。推奨の組み合わせは `set -Eeuo pipefail` ＋ `trap 'echo "Error at line $LINENO, exit $?" >&2' ERR`。
• `trap EXIT` はスクリプトの成功・失敗問わず実行されるためクリーンアップ処理の置き場として最適（try-finally相当）。デバッグには `set -x`（xtrace）を追加すると各コマンド実行前に stderrへ出力される。

**git non-fast-forward push と rebase の操作順序**
• `git pull --rebase` 実行前にunstagedな変更（`git add` 前のファイル変更）が存在するとリベースがブロックされる。自動化スクリプトでは `git pull` → ファイル操作 → `git add` → `git commit` → `git push` の順が鉄則。
• Git 2.34以降では、グローバルで `pull.ff only` ＋ リポジトリ単位で `pull.rebase true` の組み合わせはトラブルになりうる。同リポジトリで `git config pull.ff true` も追加設定することで回避できる。
• `git pull --ff-only` はfast-forward可能な場合のみ成功し、ダイバージした場合はエラーで止まる。自動化スクリプトで予期しないマージコミットを防ぐ安全策として有効。
