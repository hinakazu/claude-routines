**cronとPATH管理**
• cronジョブは非インタラクティブ・非ログインシェルで動作するため、.zshrcや.bash_profileが読み込まれず、カスタムPATHが失われる。crontab冒頭で `PATH=/usr/local/bin:/usr/bin:/bin` のように明示的に指定するのが最も確実な対策。
• `/usr/local/sbin:/usr/local/bin` 等は多くのLinux環境のcronデフォルトPATHに含まれないため、`which claude` で確認した絶対パスをスクリプト内で使うかcrontabに追記する。
• macOSではlaunchdがcronの代替だが、launchdも `/usr/local/bin` をデフォルトPATHに含まないことが多い。ラッパースクリプト内でPATHを明示するか、plistの `EnvironmentVariables` キーで設定するのが安全。
• 診断手順：① crontabに `* * * * * env > /tmp/cron_env.txt` を追加してenv確認 → ② スクリプト先頭に `set -x` とPATH出力を追加 → ③ 絶対パスで修正。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体では「どこで失敗したか」が分からない。`trap 'echo "ERROR at line $LINENO, exit code $?" >&2' ERR` を組み合わせることで、失敗行と終了コードをログに残せる。
• `local var=$(command)` はassignmentがexit 0を返すため、コマンド失敗でも `set -e` が発火しない落とし穴がある。`local var; var=$(command)` の2行に分けることで回避できる。
• `set -euo pipefail` の組み合わせが現代的なベストプラクティス。`-u` は未定義変数の使用を禁止、`-o pipefail` はパイプ途中の失敗を検知する。`trap EXIT` で一時ファイルのクリーンアップも確実に行う。
• Bash 4.4以上では `set -E` + `inherit_errexit` でサブシェル内でも `set -e` を継承できる。コマンド置換内の失敗が握りつぶされなくなる。

**git non-fast-forward pushと自動化スクリプトの操作順序**
• 自動化スクリプトでgit操作する場合の安全な順序：① `git stash -u`（未コミット変更を退避）→ ② `git pull --rebase origin <branch>`（リモートの差分取り込み）→ ③ ファイル操作 → ④ `git add` → ⑤ `git commit` → ⑥ `git push`。「ファイルを置いてからpull」の順序が non-fast-forward エラーの根本原因。
• `git pull --rebase` はmerge commitを作らずリニアな履歴を保てるため、複数箇所からpushが起きる自動化スクリプト向きの戦略。競合が起きると `git rebase --continue` が必要になるため、スクリプトでは `|| git rebase --abort` 時のロールバック処理も実装する。
• push前の防衛コード：`git fetch origin && git log HEAD..origin/<branch> --oneline` で差分を確認し、差分がある場合はrebaseを実行してからpushする。
