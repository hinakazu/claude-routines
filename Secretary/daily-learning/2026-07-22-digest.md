**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` / `.bash_profile` を一切読み込まないため、`/usr/local/bin` や homebrew のパスは未登録のまま。コマンドがexit 127で無音終了する原因はほぼ常にこれ。
• crontabの先頭行で `PATH=/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin` と `SHELL=/bin/bash` を明示するか、スクリプト内でフルパス（例: `/opt/homebrew/bin/claude`）を使うのが最も確実。
• macOSではAppleが推奨するlaunchdへの移行が長期的解決策。スリープ中のジョブ漏れがなく、SIPとのアクセス制御も統合されており、ファイル変更トリガー等の拡張イベントにも対応。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単独では、`if`条件内・パイプライン・`||`/`&&`チェーンでのコマンド失敗時は終了しない。`((count--))` のような算術式が結果ゼロを返すと意図せず終了するケースも存在する。
• 推奨の組み合わせは `set -euo pipefail`。`-u` は未定義変数の参照をエラーに、`-o pipefail` はパイプライン途中の失敗を確実に捕捉する。
• `trap 'echo "Error on line $LINENO: exit $?" >&2' ERR` と `trap cleanup EXIT` を組み合わせると、どの行で何のコードで失敗したかがログに残り、無音終了を完全に排除できる。`set -eE` を使うとtrap ERRがサブシェル内にも伝播する。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトで複数箇所からpushが発生する場合、`git pull --rebase` を「ファイル変更より前に」実行しなければ、unstaged changesが残った状態でrebaseがブロックされる。正しい順序は `pull --rebase` → 変更生成 → `add` → `commit` → `push`。
• `git config pull.rebase true` でリポジトリのデフォルトをrebaseに統一すると、スクリプト内で毎回 `--rebase` フラグを指定し忘れるリスクを排除できる。
• push失敗時は自動でpullして再試行する仕組みが自動化では必須。GitHubのgit-auto-commit-actionはnon-fast-forwardエラー検知 → pull → retry のパターンを実装しており、同種の処理を自前スクリプトに取り込む参考になる。
