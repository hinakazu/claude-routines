**cronとPATH管理**
• cronはログインシェルの `.zshrc` や `.bash_profile` を読まず、最小限のPATH（`/usr/bin:/bin` 等）のみで起動する。そのためclaudeやnvmなどユーザーインストールのコマンドが見つからずexit 127でサイレント終了する。
• 最も確実な解決策は①crontabの先頭に `PATH=/usr/local/bin:/usr/bin:/bin:/Applications/cmux.app/Contents/Resources/bin` と明示する方法と、②ラッパースクリプトで `source /etc/profile && source ~/.bash_profile` してから実コマンドを呼ぶ方法の2択。
• `BASH_ENV=/path/to/env.sh` をcrontabに設定するとbashが各ジョブ前にその環境スクリプトを自動実行する。複数ジョブ共通の設定を一元管理できる。

**bash `set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は if/elif/while/until条件内、`&&`/`||` リストの途中、パイプラインの途中コマンドでは発火しない。これらのコンテキストで失敗しても黙って続行するため、デバッグが困難になる。
• コマンド置換 `$(...)` 内のエラーは親シェルに伝播しない（`inherit_errexit` シェルオプションを設定しないと `set -e` が無効）。サブシェル内の失敗をキャッチするには `result=$(cmd) || handle_error` と明示的に書く必要がある。
• 実用的なテンプレートは `set -euo pipefail` と `trap 'echo "Error on line $LINENO: $BASH_COMMAND (exit $?)" >&2' ERR` の組み合わせ。`$LINENO`・`$BASH_COMMAND`・`$?` でどのコマンドが何番行で何のコードで失敗したかをログに残せる。

**git non-fast-forward と自動化スクリプトでの操作順序**
• non-fast-forwardエラーはローカルとリモートの両方に相手が持っていないコミットが存在するとき発生する。自動化スクリプトで起きやすいのは「他箇所からpushが発生→ローカルがcp/commit→push」というタイミングの競合。
• 正しい順序は `git pull → ファイル変更 → git add → git commit → git push`。変更作業の前にpullすることで、リモートの最新状態を取り込んだ上でコミットが積まれる。
• Git 2.6以降では `git config pull.rebase true` と `git config rebase.autoStash true` を設定すると、uncommitted changesがある状態でもpullが安全に実行される（自動的にstash→rebase→unstashを行う）。自動化スクリプトへの組み込みに有効。
