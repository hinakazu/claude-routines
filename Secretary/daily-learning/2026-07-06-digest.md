**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` / `.bash_profile` を読み込まず、`PATH=/usr/bin:/bin` のみが設定される。`/Applications/...` など独自パスは未登録でコマンドがexit 127でサイレント終了する。
• 最もシンプルな解決策はcrontabの先頭に `PATH=/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin:/opt/homebrew/bin` を追記すること。または各コマンドをフルパスで記述する。
• macOSではApple公式推奨の `launchd` がcronの代替。plistの `EnvironmentVariables` セクションでPATHを明示的に宣言でき、スリープ/ウェイク時の実行スキップ問題も解消される。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` の典型的な罠: `local var=$(command)` は `local` の終了コードが0になるため失敗が隠蔽される。宣言と代入を分離して `local var; var=$(command)` と書くのが正解。
• パイプラインの末尾コマンドが成功すれば途中の失敗が無視される問題は `set -o pipefail` で解決。コマンド置換内の `set -e` 無効化は `shopt -s inherit_errexit`（Bash 4.4+）で対処。
• 推奨パターン: スクリプト冒頭に `set -euo pipefail; shopt -s inherit_errexit` を組み合わせ、`trap 'echo "ERROR: line $LINENO, exit $?" >&2' ERR` でエラー時にファイル・行番号・終了コードをログに残す。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーは「リモートがローカルより進んでいる状態でpushしようとしたとき」に発生する。自動化スクリプトでは `pull → ファイル変更 → add → commit → push` の順序を守ることが最重要。
• `git pull --rebase` はマージコミットを作らずリモートの変更をローカルコミットの下に差し込む。`git config --global pull.rebase true` でデフォルト化すると一貫性が保てる。
• 2025-2026年の調査では、rebaseワークフローを採用したチームはマージコンフリクトが30〜40%減少。自動化スクリプトでは `git stash → git pull --rebase → git stash pop` のパターンで unstaged changes があっても安全にpullできる。
