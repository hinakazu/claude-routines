**cronとPATH管理**
• cronはユーザーの .zshrc/.bash_profile を読み込まず、PATH は /usr/bin:/bin など最小限のみ。手動実行では動くのにcronで exit 127 になる典型的な罠は「コマンドが見つからない」。スクリプト冒頭か crontab ファイル上部に `PATH=/usr/local/bin:/usr/bin:/bin:...` を明示することで解消する。
• macOSでは cron より **launchd** が推奨。cron はMac がスリープ中のジョブをスキップするが、launchd は起動後にキャッチアップ実行する。launchd の plist では `EnvironmentVariables` キーに PATH を明示し、Apple Silicon Mac では `/opt/homebrew/bin` を必ず追加する。
• macOS Mojave以降はプライバシー保護強化により、Desktop/Documents/Downloads 等のディレクトリへのバックグラウンドアクセスが無音でブロックされる。cronもlaunchdも同様に影響を受けるため、「フルディスクアクセス」権限を付与する必要がある。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は条件式（`||`/`&&`）の中にあるコマンド失敗をキャッチしない。`cmd || echo "failed"` のように書くと cmd が失敗しても終了せず、エラーが隠蔽される。
• パイプのエラーは `set -e` 単体では捕捉できない。`false | true` は終了コード 0 になるため、`set -o pipefail` をセットで使う必要がある。
• 算術演算 `((count++))` は count=0 のとき戻り値が 0（偽）となり `set -e` が即終了する。回避策は `count=$((count + 1))`。
• 本番スクリプトの定石は `set -euo pipefail` + `trap 'echo "[$(date)] ERR on line $LINENO, exit $?" >> /path/to/error.log' ERR`。これでどの行で何の終了コードで失敗したかをログに残せる。
• スクリプト終了時の共通クリーンアップには `trap cleanup EXIT` を使う。正常終了・エラー終了・Ctrl+C いずれでも確実に実行される。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーの本質は「リモートがローカルより先に進んでいる」状態。`git push` が拒否されるのはリモートのコミットを上書き・消去しないための保護機構。
• 自動化スクリプトでの正しい操作順序: **`git pull --rebase` → ファイル変更 → `git commit` → `git push`**。変更後に `git pull --rebase` を行うと unstaged changes がある場合にrebaseがブロックされる。
• `git pull --rebase` はリモートのコミット上にローカルコミットをリプレイするため、マージコミットが発生せず線形履歴になる。CI/CDログが追いやすく `git bisect` との相性も良い。
• 共有ブランチ（main/master）への force push は他者の変更を消す危険があるため禁止。force push は自分だけの feature ブランチに限定する。
