**cronとPATH管理**
• cronは起動時に `/usr/bin:/bin` のみのミニマル環境で動作し、`.zshrc` や `.bashrc` は一切読み込まれない。`/usr/local/bin` や Homebrew の `/opt/homebrew/bin`（Apple Silicon Mac）は未登録となり、コマンドが exit 127 でサイレント失敗する。
• 根本解決策は3通り: ①crontab先頭に `PATH=/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin` を1行で明示（export不要・変数展開不可）、②スクリプト内で絶対パスを使う（`which コマンド名` で確認）、③ラッパースクリプトで環境をセットアップしてから本体を呼ぶ。
• デバッグには `env -i HOME=$HOME PATH=/usr/bin:/bin /bin/sh -c '/path/to/script.sh'` でcron環境を再現する。macOSではAppleがcronをdeprecatedとし、launchdへの移行を推奨。launchdは `EnvironmentVariables` キーでPATHを明示する必要があり、スリープ中のジョブを起動後に遅延実行する（cronはスキップする）利点がある。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `if`/`elif`/`while` の条件部、`&&`/`||` リスト途中のコマンド、パイプライン最終以外のコマンドでは発動しない。コマンド置換（`$(...)`）内でも継承されないため、サブシェル内のエラーが親スクリプトをサイレント終了させる。算術展開 `((count--))` は結果が0のとき終了コード1を返し予期せず終了する（`((count--)) || true` で回避）。
• 完全なエラーハンドリングには `set -euo pipefail` と `trap 's=$?; echo "$0: Error on line $LINENO: $BASH_COMMAND (exit: $s)"; exit $s' ERR` の組み合わせが推奨される。`set -E`（errtrace）を加えるとコマンド置換内でもERRトラップが継承。コマンド置換内の `errexit` 継承には `shopt -s inherit_errexit`（Bash 4.4+）が必要。
• クリーンアップ処理は `trap finish EXIT` で設計すると、`set -e` による予期せぬ終了時でも必ず実行される。`set -e` と `pipefail` はサイレントなので、必ず `trap ERR` でログを出力する仕組みを追加するのが鉄則。

**git non-fast-forward と自動化スクリプトの操作順序**
• non-fast-forwardエラーは「リモートに自分のローカルにないコミットが存在する状態でpushしようとした時」に発生する。自動化スクリプトで `cp → commit → push` の順序をとると、pull前にファイルが更新されてunstaged changesが残り、`git pull --rebase` がブロックされる。
• 正しい操作順序は `pull → ファイル操作（cp等）→ add → commit → push`。pullを最初に実行してローカルを最新化してから変更を加えることで競合を防ぐ。`git pull --rebase --autostash` を使うと未コミット変更を自動 stash → rebase → unstash してくれる。`git config --global pull.rebase true` でrebaseをデフォルト化も可能。
• すでにリモートにpushしたコミットへのrebase適用はhistoryが書き換わるため共有ブランチには使ってはならない。やむを得ない場合は `git push --force-with-lease` で他者の push を上書きするリスクを最小化できる。
