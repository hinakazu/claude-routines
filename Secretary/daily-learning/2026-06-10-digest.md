**cronとPATH管理**
• cronジョブはユーザーの `.zshrc`/`.bashrc` を読み込まず、PATH は `/usr/bin:/bin` 程度しか設定されない。Homebrew やユーザーローカルのバイナリ（例: `/opt/homebrew/bin`, `/Applications/cmux.app/Contents/Resources/bin`）は使えず exit 127 でサイレント失敗する
• スクリプト冒頭または crontab 先頭で PATH を明示指定するのが最善。`PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin` と書く。`PATH=$PATH:/custom/bin` は cron が変数展開しない実装があるため NG
• macOS では Apple が cron を非推奨とし launchd を推奨。launchd は Mac がスリープ中にスケジュール時刻を過ぎた場合、起動後に遅延実行してくれる（cron はスキップする）

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `&&` / `||` の後ろ、`if` の条件節、`!` で反転したコマンドでは失敗しても終了しない。コマンド置換 `$(fail_cmd)` 内のエラーもデフォルトでは伝播しない（`inherit_errexit` オプションで解消可能）
• `set -eEo pipefail` + `trap 'echo "ERR line $LINENO exit $?" >> /path/error.log' ERR` の組み合わせがベスト。`-E`（errtrace）により関数・サブシェル内でも ERR トラップが継承される
• 算術展開 `((count--))` は結果が 0 のとき終了コード 1 を返し予期せず終了する。`((count--)) || true` で明示的に逃げる

**git non-fast-forward push と rebase の操作順序**
• `git pull --rebase` はアンステージな変更がある状態で "cannot pull with rebase: You have unstaged changes" となりブロックされる。自動化スクリプトでは `git pull --rebase --autostash` を使うと未コミット変更を自動 stash → rebase → unstash してくれる
• 自動化での安全な操作順序: `git fetch → (必要なら git stash) → git rebase origin/main → ファイル操作 → git add → git commit → git push`。ファイルをコピーしてから pull するとアンステージ変更になりブロックされる
• 公開済みブランチ（main など）の rebase は禁止。やむを得ず push し直す場合は `git push --force-with-lease` を使うことで他者の push を上書きするリスクを最小化できる
