**cronとPATH管理**
• cronはログインシェル（`.bashrc`/`.zshrc`）を一切読まないため、デフォルトPATHは `/usr/bin:/bin` のみ。`/usr/local/bin` や `/Applications/cmux.app/.../bin` などユーザー固有パスは自動では含まれない。スクリプト冒頭で `export PATH=/usr/local/bin:/usr/bin:/bin:$PATH` を必ず明示すること。
• cronでは `PATH=$PATH:/foo` のような変数展開が実装によって動作しないため、必ず固定値で完全パスを指定する。`#!/bin/bash -l` シェバンでログインシェルとして実行する方法もあるが、副作用（ログイン時スクリプトの全実行）があるため本番環境では慎重に。
• macOSでは cron より **launchd** が公式推奨。`~/.plist` の `EnvironmentVariables` キーに `<key>PATH</key><string>/usr/local/bin:/usr/bin:/bin</string>` と明示することで確実にパスが通る。launchd はマシンがスリープ中でも復帰時にジョブを実行する点でもcronより優れる。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `if`/`elif`/`while` の条件式、`&&`/`||` チェーンの中間コマンド、パイプラインの非末尾コマンドでは**発火しない**。例えば `cmd1 && cmd2` で `cmd1` が失敗しても `set -e` はスクリプトを終了させない（`cmd2` の後でのみ評価される）。
• 推奨パターンは `set -eEo pipefail` ＋ `trap 'echo "Error at line $LINENO, exit $?" >> /tmp/script.log' ERR` ＋ `trap cleanup EXIT` の組み合わせ。`-E`（errtrace）を付けることでERRトラップが関数・サブシェル・コマンド置換にも継承される。
• `inherit_errexit` オプション（bash 4.4+）を有効にすると、コマンド置換 `$(...)` の内部でも `errexit` が機能する。自動化スクリプトでは失敗を必ずログファイルに書くか、終了コードを `|| { echo "FAILED $?" >&2; exit 1; }` で明示的にキャッチする設計にする。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトで `cp ファイル → commit → git pull --rebase → push` の順は危険。`pull --rebase` 実行時に unstaged changes があるとリベースがブロックされる。正しい順序は **`git stash → git pull --rebase → git stash pop → ファイルコピー → git add → commit → push`**。
• `git config pull.rebase true` ＋ `git config rebase.autoStash true` をリポジトリに設定すると、`git pull` が自動的に stash/unstash を処理してくれる。スクリプト内で明示的な stash を書かなくてよくなり、操作順序ミスによるブロックを防げる。
• `git stash` はスタックに積まれるため冪等性がなく、スクリプトが途中で失敗・再実行されると stash が溜まっていく。本番自動化スクリプトでは `git stash list` でスタック数を確認するか、`autoStash` 設定に委ねる方が安全。`git pull --ff-only` はローカルコミットがない場合のみ動作し、安全な代替手段になる。
