**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` や `.bash_profile` を読み込まず最小限のPATHで起動するため、`/opt/homebrew/bin` 等のカスタムパスにあるコマンドがexit 127で無音終了する。対策は3つ：crontab冒頭に `PATH=/usr/local/bin:/opt/homebrew/bin:...` を明示、shebangを `#!/bin/bash -l`（ログインシェル）にする、または冒頭で `source ~/.bash_profile` を呼ぶ。
• macOS Sequoia以降ではcronデーモン自体が起動していないケースがあり、ジョブがサイレントにスキップされる。Appleの公式後継はlaunchdで、plistファイルの `<key>EnvironmentVariables</key>` ブロックでPATHをジョブ単位で宣言でき、シェル設定に依存しない確実な管理が可能。
• launchdの追加メリット：スリープ復帰後の実行補完、`launchctl list` による状態確認、`KeepAlive` や `StartInterval` 属性によるcronにない細かい制御が可能。cronと違いジョブが失敗しても記録が残る。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -euo pipefail` の3点セットが基本。`-e` で失敗即終了、`-u` で未定義変数参照を検出、`-o pipefail` でパイプライン途中の失敗を捕捉。これだけでは原因特定が困難なので `trap 'echo "Error on line $LINENO: $BASH_COMMAND (exit $?)" | tee -a /var/log/script.log' ERR` を組み合わせて行番号・失敗コマンド・終了コードをログに残す。
• 主な落とし穴3つ：① `(( i++ ))` が0を返すと `set -e` でスクリプトが終了 → `(( i++ )) || true` で回避、② `local var=$(command)` ではコマンドの終了コードが捨てられ `set -e` が発火しない → `local var; var=$(command)` と宣言・代入を分離、③ERRトラップはデフォルトでサブシェル・関数に継承されない → `set -E` を追加して伝播させる。
• デバッグ3段階：静的解析に `shellcheck script.sh`、実行トレースに `bash -x script.sh`、エラー発生行特定に `trap` によるログ出力。「手動では動くのにcronで動かない」問題はこの順で当たれば短時間で特定できる。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーの根本原因：ローカルとリモートの両方に独立したコミットがある「diverged」状態。cronジョブと手動操作が同じブランチにpushする自動化スクリプトで特に頻発する。
• 正しい操作順序：`git stash -q` → `git pull --rebase origin main` → `git stash pop -q` → `git add` → `git commit` → `git push`。`--rebase` はローカルコミットをリモートの最新履歴の上に再適用し、マージコミットなしの線形履歴を保てる。
• 自動化スクリプトの防衛パターン：① `git diff --quiet && git diff --cached --quiet` で未コミット変更を検出してからstash、② `git pull --rebase` 失敗時は `git rebase --abort` で安全にロールバック、③ `set -e` と組み合わせてrebase失敗時にスクリプトを即停止させ、次のコマンドで状態を壊さない。
