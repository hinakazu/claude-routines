**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` や `.bash_profile` を読み込まないため、デフォルトPATHは `/usr/bin:/bin` のみになる。`/opt/homebrew/bin` 等のカスタムパスは見えず、`claude` コマンドが exit 127 で無音終了する典型的な罠。
• 対策は3通り：①crontabの冒頭に `PATH=/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin` を直書き、②スクリプトのシバン行を `#!/bin/bash -l`（ログインシェル）にする、③`BASH_ENV=/path/to/env.sh` をcrontabに設定してスクリプト実行前に環境変数ファイルを読み込む。
• macOSではAppleがcronをdeprecatedとしておりlaunchdを推奨。launchdでは `EnvironmentVariables` キーにPATHを明示的に記載でき、Apple Silicon環境では `/opt/homebrew/bin` の追記が必須。またlaunchdは `~` を展開しないため `/Users/yourname/` の絶対パス指定が必要。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e`（errexit）は一見安全だが、パイプライン・コマンド置換・`if`文・`&&`/`||` の中ではエラーを無視するという重大な例外がある。特に `VAR=$(failing_command)` はエラーで終了するがログが残らず、原因調査が困難になる。
• より堅牢な設計は `set -euo pipefail` を組み合わせた上で `trap 'echo "ERROR at line $LINENO: $BASH_COMMAND" >&2' ERR` を設置する方法。`set -E`（errtrace）を加えることで関数・サブシェル内でもERRトラップが継承される。Bash 4.4以降の `inherit_errexit` オプションはコマンド置換でのerrexitを補完する。
• 実運用では「`set -e` に完全に頼らず、重要なコマンドの終了コードは明示的にチェックする」という考え方が2025年以降のコンセンサス。自動化スクリプトでは失敗時にログファイルへ書き出す `exec 2>>logfile` の併用が有効。

**git non-fast-forward pushと操作順序**
• `non-fast-forward` エラーはローカルとリモートが diverge（分岐）した状態でpushしようとすると発生。複数箇所からpushする自動化スクリプトでは常に起こりうる。正しい操作順序は「①fetch/pull → ②ファイル操作 → ③add/commit → ④push」で、ファイルをコピーしてからpullすると未コミット変更がrebaseをブロックする。
• `git pull --rebase` は履歴をきれいに保てるが、すでにリモートにpushした変更に適用するとコラボレーターの履歴を書き換えるリスクがある。共有ブランチでは `git pull`（mergeモード）が安全。自動化では `git pull --rebase origin main` をcommit前に必ず実行し、rebaseが失敗したらスクリプトを中断してアラートする設計にする。
• 2025年のgitベストプラクティスでは、自動化スクリプトに pre-push フックを追加してテスト・lint・秘匿情報スキャンを実行し、通過した場合のみpushする「ゲートウェイ型」設計が推奨される。
