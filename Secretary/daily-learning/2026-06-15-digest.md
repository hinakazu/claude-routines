**cronとPATH管理**
• cronジョブはユーザーの.zshrc/.bash_profileを読み込まないため、ユーザー定義のPATHが使えずexit 127でサイレント失敗する。スクリプト冒頭に `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` と明示するか、`#!/bin/bash -l` のlogin shellシバンを使うことで防げる。
• crontabで `PATH=$PATH:/path` のように変数展開を使うのは危険。実装によっては展開されずリテラル文字列になり、/usr/binなど重要なパスが失われる。本番では明示的なフルパス列挙かラッパースクリプトが安全。
• macOSではAppleがcronを非推奨とし、launchdを推奨。plistのEnvironmentVariablesキーで環境変数を明示設定できる。~/Library/LaunchAgents/ でテストし、問題なければ/Library/LaunchDaemons/ に移す。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体ではパイプライン・条件式・サブシェル内のエラーを検出できず、意図せずスキップされる。必ず `set -euo pipefail` の3点セットを冒頭に記述する。-u は未定義変数をエラーとし、-o pipefail はパイプ内失敗を捕捉する。
• `trap 'echo "Error at line $LINENO" >> /tmp/error.log' ERR` でエラー発生行番号をログに残す。`trap cleanup EXIT` と組み合わせて、正常・異常どちらの終了でもクリーンアップを保証できる。
• サブ関数内でもERRトラップを継承させるには `set -E`（errtrace）が必要。`set -eEuo pipefail` がより堅牢な組み合わせ。本番スクリプトでこのパターンを欠くと60%以上がサイレントなデータ損失や不整合状態に陥るという報告がある。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーはローカルとリモートのブランチが分岐したときに発生する。複数箇所から自動pushするリポジトリでは常に起こりうる。自動化スクリプトでの正しい操作順序は「stash → pull --rebase → stash pop → commit → push」。
• `git pull --rebase` はリモートの変更をベースに自分のコミットを積み直すため、マージコミットが発生せず履歴がクリーンに保たれる。スクリプトではrebase中のコンフリクトを検出したら `git rebase --abort` してエラーログを残す設計にすること。
• コミット前にunstaged changesがあるとrebaseがブロックされる。`git stash` で一時退避し、pull後に `git stash pop` で復元してからstage → commitする順序を徹底する。
