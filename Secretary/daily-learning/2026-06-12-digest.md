**cronとPATH管理**
• cronはログインシェル（.zshrc等）を一切読み込まない最小環境で動作する。`/usr/local/bin` や `/opt/homebrew/bin` が未登録のままとなり、`claude` や `node` 等のコマンドがexit 127でサイレント終了する典型的な罠。
• 対策は3段階：①crontabファイル冒頭に `PATH=/usr/local/bin:/opt/homebrew/bin:/usr/bin:/bin` を直接記載、②複雑な設定はラッパースクリプトに切り出して先頭でPATHを明示、③macOSではApple公式推奨のlaunchdへ移行（`EnvironmentVariables` キーでPATHを宣言）。
• launchdはcronと異なりスリープ中に逃した実行を起動後に補完してくれるため、Mac上の自動化には原則launchdを選ぶべき。Apple Silicon MacではHomebrew用に `/opt/homebrew/bin` の追加が必須。launchdでは `~` が展開されないため `/Users/yourname/` の絶対パス記述が必要。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` には3つの代表的な穴がある：①コマンド置換（`$(cmd)`）内ではerrexitが継承されない、②`&&` や `||` リスト内のコマンドは失敗してもスクリプトを続行する、③パイプでは最後のコマンドの終了コードしか評価しない（`cmd1 | cmd2` でcmd1が失敗してもcmd2が0なら続行）。
• 推奨セットは `set -euo pipefail` の3点セット。`-u` で未定義変数参照を検知、`-o pipefail` でパイプ途中の失敗も検知できる。
• エラー発生行を記録するには `set -E`（errtrace）と `trap 'LASTLINE=$LINENO' ERR` を組み合わせる。`set -E` によりERRトラップが関数・サブシェルにも継承される。さらに `BASH_LINENO`・`BASH_SOURCE`・`FUNCNAME` 配列を使えばスタックトレース出力も実装できる。特定コマンドだけトラップを外したい場合は `set +E` → 処理 → `set -E` で区間制御が可能。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーはローカルブランチがリモートHEADの直接の子孫でない（別の箇所からpushが発生した等）ときに起きる。自動化スクリプトで `cp → commit → pull → push` の順にすると、pullの時点でunstaged changesが残っておりrebaseがブロックされる。正しい順序は `pull → cp/変更 → add → commit → push`。
• `git pull --rebase` はローカルコミットをリモートの先頭に積み直し、線形な履歴を保つ。共有ブランチでは履歴書き換えを避けるため `git pull`（マージコミット）の方が安全。fast-forwardのみ許可する `git merge --ff-only @{u}` も安全策として有効。
• pre-pushフックに `git fetch && git rebase` を仕込むことで、pushの直前に自動的にリモートの最新を取り込むことができる（HuskyやGit hooks）。CI/CDでは重いテストをpre-commitから外してパイプライン側に移し、ローカルhooksは軽量に保つのが2025年の標準的なスタイル。
