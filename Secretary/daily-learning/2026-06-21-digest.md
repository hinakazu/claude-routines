**cronとPATH管理**
• cronの実行環境は `PATH=/usr/bin:/bin` のみで `.zshrc`/`.bashrc` は読み込まれない。`/Applications/cmux.app/...` などカスタムパスが未登録となりexit 127でサイレント失敗する典型パターン。
• crontabファイルの先頭に `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/path/to/custom` を完全にハードコードで記述する（`PATH=$PATH:/custom/path` はcronで変数展開されないので使用禁止）。
• スクリプト内のコマンドはすべて絶対パス（`/usr/local/bin/claude` など）で記述する。ラッパースクリプトパターン（環境変数をセットする薄いシェルを挟む）が本番推奨で、独立テストも容易になる。
• macOSではlaunchdが推奨。スリープ復帰後のジョブ実行が可能でSIPとの統合も優れる。cronはSIPによってパーミッション問題が発生しやすい。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `if`/`while`/`||`/`&&` 内のコマンド失敗では発動しない。またコマンド置換 `$(false)` は `inherit_errexit` を設定しない限り捕捉されない → 多くのシーンで"落とし穴"になる。
• 実用的な組み合わせは `set -euo pipefail`（`-u`: 未定義変数でエラー、`-o pipefail`: パイプライン途中の失敗も検知）。さらに `set -E`（errtrace）を加えるとERRトラップが関数・サブシェル内にも継承される。
• `trap 'echo "Error at line ${LINENO}, exit code $?" >&2' ERR` を先頭に置くだけでエラー箇所が特定できる。`trap cleanup EXIT` を併用すると成功・失敗問わず後処理が確実に実行される。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーはリモートに新しいコミットが存在する状態で古いローカルからpushしようとした際に発生する。自動化スクリプトで複数箇所が同一ブランチにpushする構成で特に頻発する。
• 自動化スクリプトの正しい操作順序：① `git pull --rebase` → ② ファイル変更・コピー → ③ `git add` → ④ `git commit` → ⑤ `git push`。pull前にunstaged changesがあるとrebaseがブロックされるため、必ずpullを最初に行う。
• `git config --global pull.rebase true` でデフォルトをrebaseに統一するとfast-forwardエラーを未然に防げる。チームやCI/CDではMergifyなどの自動rebaseツールでhumanエラーを排除できる。共有ブランチへのrebaseは禁止で、ローカルのfeature branchのみ対象にすること。
