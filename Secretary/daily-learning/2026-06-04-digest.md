**Claude Code MCPツールスコープとサブエージェント制約**
• サブエージェントはデフォルトで親会話の全ツール（内部ツール＋MCPツール）を継承するが、`tools`（ホワイトリスト）または`disallowedTools`（ブラックリスト）で細粒度に制御できる。両方指定すると`disallowedTools`が先に適用される
• MCPサーバーをサブエージェント専用にスコープするには、`.mcp.json`ではなくサブエージェントの`mcpServers`フィールドにインライン定義する。こうすると親会話にはツール定義トークンが一切流れず、コスト削減になる
• `Agent`/`AskUserQuestion`/`EnterPlanMode`などのUIツールはサブエージェントに渡せない仕様。また**サブエージェントが別のサブエージェントを生成することは不可**（ネスト禁止）。これが「lazyweb は桂小五郎に渡らない」の根本理由
• セキュリティ観点では、サブエージェントを「能力を絞るために使う」のがベストプラクティス。不要なMCPサーバーを5つ接続すると毎回数千トークンの定数コストが発生する

**LIFF審査中のローカル開発バイパスパターン**
• LINE公式の`@line/liff-mock`パッケージ（LIFF SDK v2.19.0以降対応）を使えば、LIFFサーバーへの接続なしにモックデータで動作するモードを有効化できる。`liff.init({ liffId: 'liff-xxxx', mock: true })`の1行で切り替わる
• `liff.$mock.set()`でプロフィール・ログイン状態・アクセストークンなどをカスタマイズ可能。`liff.$mock.clear()`でデフォルト値に戻せる
• `NEXT_PUBLIC_USE_MOCK=true`のような環境変数と組み合わせ、`.env.local`（開発）と`.env.production`（本番）で挙動を分岐させるのが推奨パターン。APIサーバー側も`mock-id-token-for-dev`等のヘッダーで分岐させると審査待ち中でもフルスタック開発が継続できる
• Create LIFF App（LINEの公式CLI）がNext.js + LIFF環境を一発生成するため、新規プロジェクトはそこから始めると環境変数セットアップが自動化される

**カードUIのアスペクト比設計原則（ヒーロー vs 一覧）**
• Material Design 3のガイドラインでは、カードヘッダー画像のアスペクト比は4:1〜3:4の範囲で、推奨デフォルトは**2:1・4:3・16:9**の3パターン。ヒーローカードは16:9が標準、一覧カードのサムネイルは1:1（正方形）が安定する
• ヒーロー画像は「コンテンツの主役」として画面幅いっぱいの横長を使い視覚的インパクトを優先。一覧カードは左サムネイル64px正方形が安定する理由は、テキスト情報と視覚情報のバランスが50:50になるためスキャン性が高い（Klook・Expedia等が採用）
• グリッド内の一貫性が最重要。同一コンテキストのカードは同じアスペクト比に揃えないと視覚的な"ずれ"が生じる。2026年のBentoグリッドトレンドは非対称比率を意図的に使うが、それは明示的なデザイン意図がある場合のみ
