**MCP ツールスコープとサブエージェント設計**
• サブエージェントはデフォルトで親の内部ツール・MCPツールを継承するが、`tools`（許可リスト）か`disallowedTools`（拒否リスト）で絞り込める。両方指定した場合は`disallowedTools`が先に適用される。
　🔗 https://code.claude.com/docs/en/sub-agents
• `Agent`・`AskUserQuestion`・`EnterPlanMode`など UI 操作系ツールは設計上サブエージェントに渡らない。lazyweb のような MCP ツールも同様の制約が起きうるため、親エージェントが直接実行して結果だけ渡す構成が必要になる。
　🔗 https://code.claude.com/docs/en/sub-agents
• サブエージェント定義の `mcpServers` フィールドで特定の MCP サーバーをサブエージェント専用にスコープできる。これにより親会話のコンテキストにツール定義が読み込まれず、トークン消費を抑えられる。
　🔗 https://code.claude.com/docs/en/sub-agents

**LIFF 審査中のローカル開発バイパス**
• `@line/liff-mock`（LIFF SDK v2.19.0 以降対応）を使えば LINE クライアントなしでローカル開発が可能。`liff.use(new LiffMockPlugin())` 後に `liff.init({ liffId: '...', mock: true })` でモックモードを有効化する。
　🔗 https://github.com/line/liff-mock
• モックモードでは `liff.getProfile()` などの LIFF API がモックデータを返し、実サーバー通信が不要になる。`liff.$mock.set()` でプロフィール名や画像 URL を任意のデータに上書きできる。
　🔗 https://github.com/line/liff-mock
• 審査待ち期間に `NEXT_PUBLIC_USE_MOCK=true` などの環境変数で認証をバイパスするカスタム実装は、`@line/liff-mock` という LINE 公式の標準手段で置き換えられる。本番コードを汚染せず切り替えられる点が優位。
　🔗 https://github.com/line/liff-mock

**カード UI のアスペクト比設計原則**
• Google の Android TV UI ガイドラインでは、動画・写真コンテンツには **16:9**、アバター・ロゴには **1:1**（正方形）、書籍・ポスターには **2:3**（縦長）の 3 種類が基本比率として定義されている。
　🔗 https://developer.android.com/design/ui/tv/guides/components/cards
• ヒーローカードは画面幅を活かした横長（16:9）が基本で、一覧カードのサムネイルは 1:1 が複数並べる際の視認性を保つのに適している。今回の EventHeroCard と ChallengeList の使い分けと完全に一致する。
　🔗 https://developer.android.com/design/ui/tv/guides/components/cards
• コンパクトカードで背景画像上にテキストを重ねる場合は Scrim（黒グラデーションオーバーレイ）が必須。テキストの視認性を確保するこの手法はモバイル UI 全般で広く使われる標準パターン。
　🔗 https://developer.android.com/design/ui/tv/guides/components/cards
