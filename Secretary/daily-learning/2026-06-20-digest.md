**cronとPATH管理**
• cronはユーザーの`.zshrc`や`.bash_profile`を読み込まないため、カスタムパスが未登録になりexit 127で無音終了する。スクリプト冒頭で`PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`を明示指定するか、`#!/bin/bash -l`のログインシェルshebangを使って回避できる。
• crontabファイル内に直接`VARIABLE=value`形式で変数宣言することで、そのcronジョブ実行時の環境に変数を追加できる最もシンプルな方法。
• macOSでは2005年からAppleがcronの代替としてlaunchdを推奨。launchdはスリープ中に見逃したジョブをウェイク後に実行でき、PATHも`EnvironmentVariables`キーで明示設定できるため、Mac上の自動化はlaunchdへ移行するのが現時点でのベストプラクティス。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e`は`&&`/`||`チェーン内コマンド・`if`/`elif`の条件部分・デフォルト設定の`$()`コマンド置換では発動しない。`set -euo pipefail`に加えて`set -E`（errtrace）と`shopt -s inherit_errexit`を追加することでサブシェルやコマンド置換にも伝播させられる。
• `trap 'handle_error "$?" "${BASH_COMMAND}" "${FUNCNAME[0]:-main}(${BASH_SOURCE[0]}:${LINENO})"' ERR`のようなカスタムERRハンドラーで、失敗コマンド名・行番号・終了コードをセットでログに残せる。これによりサイレント終了の原因追跡が格段に楽になる。
• パイプラインの部分失敗は`set -e`だけでは捕捉できず`-o pipefail`が必須。例：`cat file.txt | grep pattern`でcatが失敗しても`-o pipefail`なしでは握りつぶされる。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーの本質は「ローカルのコミット履歴がリモートから分岐した状態でpushしようとすること」。自動化スクリプトでは`pull → ファイル操作 → commit → push`の順を徹底し、pullを最初に行うことで既存のリモート変更を取り込んでから自分の変更を重ねる。
• `git config --global pull.rebase true`でデフォルト動作をrebaseに変更すると、マージコミットなしにローカル変更をリモートの最新コミットの上に再適用でき、履歴が直線的に保たれる。
• rebased後のpushには`--force-with-lease`を使うと、自分がfetchしていない新しいリモートコミットがある場合にpushを拒否するため`--force`より安全。共有ブランチへのforce pushは絶対禁止。
