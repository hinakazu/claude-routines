**cronとPATH管理**
• cronジョブは `.zshrc` や `.bash_profile` を読み込まないため、デフォルトのPATHは `/usr/bin:/bin` のみ。Homebrewの `/opt/homebrew/bin` やカスタムツール（`claude`コマンド等）が見つからずexit 127でサイレント失敗する。
• 対策は3つ：①crontabファイルの先頭に `PATH=/usr/local/bin:/usr/bin:/bin:...` を明示記述、②スクリプト冒頭で `export PATH=...` を設定、③絶対パスでコマンドを指定（最も確実）。
• macOSではcronより `launchd` の方がPATH管理が柔軟。plistファイルに `EnvironmentVariables` キーで環境変数を直接定義でき、スリープ/ウェイク対応やファイル変更トリガーなど機能も豊富。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` には重大な落とし穴が3つある：①`||`/`&&` 条件に続くコマンドは失敗してもトリガーされない、②`local var=$(失敗コマンド)` は `local` の終了コードがマスクして失敗を隠蔽する、③`((count++))` の値が0の場合にスクリプトが意図せず終了する。
• 推奨設定は `set -euo pipefail`：`-e`（コマンド失敗で即終了）、`-u`（未定義変数の参照をエラー）、`-o pipefail`（パイプライン中のどのコマンド失敗もキャッチ）の3つを組み合わせる。
• エラー発生箇所を特定するには `trap 'echo "Error at line $LINENO: $BASH_COMMAND" >> error.log' ERR` を使う。`$BASH_COMMAND` で失敗コマンドを、`$LINENO` で行番号をログに残せる。`set -x` と組み合わせると全コマンドのトレースも可能。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーは「リモートブランチがローカルのcommitの祖先でない」状態で発生。自動化スクリプトで複数箇所からpushが競合するとこの状態になる。
• 正しい操作順序は `git pull --rebase → ファイル操作 → git add → git commit → git push`。ファイルをコピーしてからpullすると未コミット変更が存在し、rebaseがブロックされる。
• `--force` ではなく `--force-with-lease` を使う習慣が重要。`--force` は他者の変更を上書きするリスクがあるが、`--force-with-lease` はリモートに予期しない変更がある場合にプッシュを拒否する安全策がある。自動化スクリプトでは `git stash -u` で未追跡ファイルも含めて一時保存してからpullし、操作後に `git stash pop` するパターンも有効。
