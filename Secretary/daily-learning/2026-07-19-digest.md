**cronとPATH管理（cron vs launchd on macOS）**
• cronはユーザーの`.zshrc`/`.bash_profile`を読み込まないため、`/opt/homebrew/bin`や独自パスが未登録になり`command not found`やexit 127で無音終了する。crontabの先頭に `PATH=/usr/local/bin:/opt/homebrew/bin:/usr/bin:/bin` を明示する、またはスクリプト内でフルパス（`/opt/homebrew/bin/node`等）を使うのが定番の防衛策。
• macOSではAppleがcronの代替として**launchd**を推奨。launchctl + `.plist`ファイルで管理し、スリープ/起動時の missed execution 検出・再実行、SIP（System Integrity Protection）との統合などcronにない強みがある。ただしlaunchdも環境変数は最小限のため、`.plist`の `EnvironmentVariables` キーに `PATH` を明示（Apple Siliconは `/opt/homebrew/bin` 追加必須）。
• cronは「一時的/Linux共通スクリプト/失敗許容」向け、launchdは「macOS常駐/スリープ対応/クリティカルジョブ」向けと使い分けると迷いが減る。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e`（errexit）はコマンドが非ゼロを返した瞬間にスクリプトを終了させるが、`||`/`&&` で繋いだ式の中・パイプラインの途中・`local var=$(failing_cmd)`の代入では **発動しない**という多数の例外がある。`((count++))` のような算術展開も最初のループでexit 0を返しスクリプトが死ぬ罠がある。
• 推奨セットアップは `set -euo pipefail`（未定義変数チェック＋パイプライン全体の失敗伝播）に加え、`set -E` でERRトラップをサブシェル・関数に継承させる。`trap 'echo "ERROR: line $LINENO, cmd: $BASH_COMMAND, exit: $?" >&2' ERR` をスクリプト冒頭に置くと失敗箇所が特定できる。
• リソース作成後すぐに `trap 'cleanup' EXIT` を登録しておくと、`set -e`で途中終了した場合も後処理が走り、ゾンビプロセスや一時ファイルが残らない。サブシェル内のtrapは引き継がれないので `$()`や`()` 内には再登録が必要。

**git non-fast-forward push と自動化スクリプトの操作順序**
• non-fast-forward エラーは「リモートにローカルが持っていないコミットがある状態でpushしようとした」ときに発生。複数箇所からpushが走る自動化スクリプトでは特に頻発する。安全な解消手順は `git fetch origin` → `git rebase origin/<branch>`（または `git pull --rebase`）→ commit → push の順序を必ず守ること。
• 自動化スクリプト内でgitを扱う際は **push前にfetchを挟む**のが鉄則。`git pull --rebase` は履歴を線形に保ちつつリモートの変更を取り込む。force pushが必要な場合は `--force-with-lease`（リモートの想定HEADと一致する場合のみ上書き）を使い、通常の `--force` は共有ブランチでは禁忌。
• rebaseファーストのワークフローを採用しているチームではマージコンフリクトが30〜40%減少するというデータがある。CI/CDパイプラインにはpush前のブランチ同期チェックを組み込むことでnon-fast-forwardを未然に防げる。
