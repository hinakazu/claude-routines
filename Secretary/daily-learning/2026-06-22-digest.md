**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` / `.bash_profile` を一切読み込まない。macOSなら `/Applications/xxx/bin` 等のカスタムパスが全滅する。対策は3層構造：①crontabの先頭に `PATH=/usr/local/bin:/usr/bin:/bin:/your/custom/path` を明示、②ラッパースクリプトを作りその中で `source ~/.bashrc` してから本体を呼ぶ、③コマンドは全て絶対パスで書く。
• macOS Sequoia（15系）ではcronがフルディスクアクセス制限の影響を受け不安定になった。Appleの推奨はlaunchd（plistファイル）への移行。launchdも `.zshrc` は読まないが、plistの `<key>EnvironmentVariables</key>` セクションでPATHを明示できる。
• crontabでは変数展開が使えない落とし穴がある。`PATH=$PATH:/home/user/bin` と書いてもリテラル文字列 `$PATH` として解釈され、既存PATHが消える。必ずフルパスを列挙すること。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` が「効かない」4つのケース：(1) `if` / `elif` の条件部分、(2) `while` / `until` のループ条件、(3) `&&` や `||` で繋いだコマンドの途中、(4) パイプの最後以外のコマンド。これらの失敗はスクリプトを止めない。
• コマンド置換（`$(...)` 内部）では ERR トラップが継承されない。サブシェルのエラーが素通りする。解決策は `set -E`（errtrace）+ `inherit_errexit` を組み合わせること：`set -E -o inherit_errexit` で、コマンド置換の中でもエラーが伝播するようになる。
• 自動化スクリプトの推奨テンプレート：`set -eEuo pipefail` + `trap 'echo "ERR: line $LINENO, cmd=$BASH_COMMAND, exit=$?" >> /var/log/myscript.log' ERR` とすることで、失敗行・失敗コマンド・終了コードをログに残せる。`-u` は未定義変数を、`pipefail` はパイプ途中の失敗を捕捉する。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーは「ローカルブランチとリモートブランチが分岐している」ときに発生する。自動化スクリプトで典型的なミス順序は `cp→commit→pull→push`。正しい順序は **pull→cp→commit→push**。pullを先にすることでローカルのHEADをリモートに揃えてから変更を積む。
• git 2.6以降は2つの設定で自動化が安定する：`git config pull.rebase true`（pull時にmergeではなくrebaseを使用）+ `git config rebase.autoStash true`（unstaged changesを自動stash→rebase→popする）。これによりダーティな作業ツリーでも pull がブロックされなくなる。
• 自動化スクリプトでのガード例：`git fetch origin main && git rebase origin/main || { echo "rebase failed" >> error.log; exit 1; }` のように fetch と rebase を分けて書くと、競合時にrebaseだけ失敗させてpushを止められる。
