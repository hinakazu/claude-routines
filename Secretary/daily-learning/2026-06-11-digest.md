**cronとPATH管理**
• cronジョブはユーザーの `.zshrc/.bashrc` を一切読み込まないため、デフォルトPATHは `/usr/bin:/bin` 等の最小限のみ。スクリプト冒頭に `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:$PATH` を明示的に記述するか、crontabファイル冒頭にPATHを定義するのがベストプラクティス。
• macOSではlaunchdが推奨。plistの `EnvironmentVariables` キーでPATHを設定でき、Sleepからの復帰後も確実に実行される（cronはSleep中にスキップされる）。
• デバッグTips: cronで動かす前に `env -i /bin/bash --noprofile --norc script.sh` でcron相当の環境を手動で再現して動作確認するとよい。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `if/while/until` 後のコマンド、`&&/||` リスト内のコマンド、パイプ途中のコマンドでは終了しない（見た目は動いているのに失敗を無視している状態）。コマンド置換 `$()` 内の失敗も無音になる。
• 推奨パターンは `set -Eeuo pipefail` ＋ `trap 'echo "ERROR at line $LINENO: $BASH_COMMAND" >> /tmp/script.log' ERR`。`-E`（errtrace）で関数やサブシェルにもERRトラップを継承させる。Bash 4.4+ では `shopt -s inherit_errexit` でコマンド置換内の失敗も捕捉できる。
• `$BASH_COMMAND` でエラーになったコマンド文字列、`$LINENO` で行番号、`caller` でファイル名を取得できるので、トラップ内で詳細ログを残すとデバッグが格段に楽になる。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでの正しい順序は「`git stash` → `git pull --rebase` → `git stash pop` → (ファイル操作) → `git add` → `git commit` → `git push`」。変更を加える前にpullするなら stash 不要で「`git pull --rebase` → (ファイル操作) → commit → push」でよい。
• `git pull --rebase` でリモートの変更を自分のコミットの前に挿入し、線形な履歴を維持。複数箇所から自動pushするリポジトリでは `git config pull.rebase true` をスクリプト内でも設定しておくと安全。
• 安全策として `git pull --ff-only` を使うと fast-forward できない状態（= 競合状態）を検知して失敗させることができ、「気づかずに上書きする」事故を防げる。
