**cronとPATH管理**
• cronはユーザーの `.zshrc`/`.bash_profile` を一切読み込まず、`/usr/bin:/bin` などの最小限PATHで起動する。`/Applications/cmux.app/Contents/Resources/bin` のようなカスタムパスが未登録になりサイレント失敗（exit 127）に直結する。
• 解決策はcrontab冒頭に `SHELL=/bin/bash` と `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/カスタムパス` を明示記載するか、スクリプト冒頭を `#!/bin/bash -l`（login shell）にする。`PATH=$PATH:/custom/bin` のような変数展開はcron実装依存で動作しないことがある。
• macOSではAppleがcronをdeprecatedとしlaunchdを推奨。launchdはスリープ中に逃したジョブを復帰後に実行できる点が優秀。macOS Mojave以降プライバシー保護強化によりcronがサイレント失敗するケースも増加している。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` はパイプライン・条件文・サブシェル内のエラーを無視する仕様がある。例：`cat /nonexistent | grep key` ではパイプ末尾のgrepが成功扱いになりスクリプトが続行する。これを防ぐには `set -o pipefail` を追加し、`set -euo pipefail`（-u: 未定義変数エラー）を組み合わせるのが2025-2026年の標準的ベストプラクティス。
• `trap 'echo "ERROR: line ${LINENO}, cmd: ${BASH_COMMAND}" >&2' ERR` を冒頭に置くと失敗行番号とコマンドがログに残り、デバッグが劇的に楽になる。`set -E` フラグを追加するとERRトラップが関数・サブシェルにも継承される。
• シグナルの使い分け：ERR は「エラー報告」用、EXIT は「クリーンアップ」用（成功/失敗問わず実行）、INT は「Ctrl+C割り込み」用。自動化スクリプトには最低限 ERR+EXIT のトラップセットを入れると障害時の原因特定が格段に速くなる。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーはローカルブランチとリモートが乖離したときに発生する。自動化スクリプトで「コピー → commit → pull → push」の順で実行すると rebase がブロックされることがある。正しい順序は `git fetch → git pull --rebase → ファイル変更 → git add → git commit → git push`。
• `git pull --rebase` はmergeコミットを作らずローカルコミットをリモートの先頭に乗せる。`git config --global pull.rebase true` で全リポジトリのデフォルト動作にできる。git 2.34+では `git config pull.ff true` でfast-forward onlyに設定可能。
• 複数箇所から同じリポジトリにpushする自動化環境（cron＋手動）では、push前に必ずfetch→rebaseを行い「自分の変更が最新リモートの上に乗っている」状態を保証することでnon-fast-forwardエラーの大部分を予防できる。
