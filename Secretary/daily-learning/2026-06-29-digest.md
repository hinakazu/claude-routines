**cronとPATH管理**
• cronジョブは `.zshrc` や `.bashrc` を読み込まないため、ユーザー環境の `PATH` は一切引き継がれない。スクリプト冒頭で `PATH=/usr/local/bin:/usr/bin:/bin:$PATH` のように明示指定するか、crontabファイルの先頭行に `PATH=...` を書くのが鉄則（`export` 不要）。
• macOSでは Apple が cron を deprecated にしており `launchd` が公式推奨。plist の `EnvironmentVariables` セクションで PATH を宣言し、Apple Silicon では `/opt/homebrew/bin` を必ず含める。launchd はスリープ中に実行できなかったジョブを wake 後に再スケジュールできるため、cron より信頼性が高い。
• コマンドはフルパスで書くのが最も確実な回避策。`/usr/local/bin/node` のように書くことでPATH依存をゼロにできる。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は `||`・`&&` の条件節、`if/elif/while/until` の判定部、パイプの途中コマンドでは発火しない。たとえば `cmd || echo "failed"` の `cmd` が失敗しても `set -e` は無視する。これが「動いているように見えてサイレント失敗する」原因。
• 推奨ベースラインは `set -Eeuo pipefail`。`-E` は ERR トラップをサブシェルに継承、`-u` は未定義変数でエラー終了、`-o pipefail` はパイプ内の最初の失敗を終了コードに伝播。
• `trap 'on_error ${LINENO} "${BASH_COMMAND}"' ERR` でエラー発生行・コマンド・終了コードをログに記録。`trap 'cleanup' EXIT` で一時ファイル削除を確実に実行。この2つのトラップを組み合わせるのが現代のベストプラクティス。

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでの正しいgit操作順序は「`git stash` → `git pull --rebase` → `git stash pop` → `git add` → `git commit` → `git push`」。unstaged changes がある状態で `git pull --rebase` を実行すると rebase がブロックされるため、先に stash で退避するのが必須。
• staged changes だけ保持して unstaged だけ退避したい場合は `git stash push --keep-index`。index（ステージ済み変更）はそのままに、ワーキングツリーの変更だけを一時退避できる。
• `git config --global pull.rebase true` でデフォルトをrebaseに設定するとリニアな履歴を保てる。force push が必要な場面では `--force` の代わりに `--force-with-lease` を使う。リモートが最新でない場合は自動アボートするため、他者のコミットを上書きするリスクを排除できる。
