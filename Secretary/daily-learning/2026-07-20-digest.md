**cronとPATH管理**
• cronのデフォルトPATHは `/usr/bin:/bin` のみ。`/usr/local/bin` は含まれず、`.zshrc` や `.bashrc` は一切読み込まれないため、手動実行では動くコマンドが cron では exit 127 でサイレント失敗する
• 最も確実な対策は crontab 先頭に `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` を明示宣言すること。または各スクリプト冒頭で `export PATH=...` を書く
• コマンドの絶対パスを `which node` / `which claude` で事前確認し、crontab 内はフルパス指定にするのも堅牢な手法。BASH_ENV 変数でロードするスクリプトを指定する方法もある

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単独では「コマンド置換 `$(...)` 内のエラーは継承しない」「`local var=$(cmd)` は代入自体が status 0 を返すためエラーをマスクする」という2大落とし穴がある
• 推奨構成: `set -euo pipefail` + `set -E`（errtrace）を組み合わせ、`trap 'echo "ERROR at line $LINENO (exit $?)" >> /var/log/script.log' ERR` でエラー発生行と終了コードを記録する
• `set -e` と `trap ERR` が競合するケース（`(( i++ ))` が 0 評価で即終了など）では、`trap ... EXIT` + `$?` で失敗を検知する方が確実。長期運用スクリプトは `2>> logfile` でエラー出力をファイルに残す

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーの根本原因は「ローカルとリモートが両方独自コミットを持ち、単純な早送りができない状態」。自動化スクリプトでは複数箇所からのpushが同時発生しやすく、この状態が頻発する
• 正しい安全手順: `git pull --rebase` → ファイル変更 → `git add` → `git commit` → `git push`。ファイルコピー後に pull すると rebase 中に unstaged changes がありブロックされる
• Git Auto Commit などの CI/CDアクションは pull を自動実行しないため、ワークフロー内で明示的に pull を先行させる必要がある。`git status` でローカルに差分がないことを確認してから pull するのが最も安全
