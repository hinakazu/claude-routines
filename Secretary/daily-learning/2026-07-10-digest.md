**cronとPATH管理**
• cronはユーザーの `.zshrc` / `.bash_profile` を読み込まず、デフォルトPATHは `/usr/bin:/bin` のみ。Homebrew等のコマンドはフルパス指定またはスクリプト冒頭で `PATH=/opt/homebrew/bin:/usr/local/bin:$PATH` と明示する必要がある
• macOSではcronよりlaunchdが推奨。スリープ中にスキップされたジョブをwake時に実行する仕組みがあり、cronはスリープ中の実行をそのままスキップしてしまう
• launchdはエージェントごとに `EnvironmentVariables` でPATHを独立設定できるため、Node・Python・Homebrew製CLIが混在する環境でも衝突しない
• Apple Silicon Macでは `/opt/homebrew/bin` を明示追加しないとHomebrewツールが全滅する（Intel Macの `/usr/local/bin` とは別パス）

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -euo pipefail` の三点セットが基本。`-u` で未定義変数をエラー検出、`-o pipefail` でパイプ途中のコマンド失敗を検出できる（`set -e` 単独では `cmd1 | cmd2` の `cmd1` 失敗を見逃す）
• `set -E`（errtrace）を追加すると `ERR` トラップが関数・サブシェル・コマンド置換にも継承され、どこで失敗しても `trap 'echo "Line $LINENO failed"' ERR` が動く
• Bash 4.4以降では `shopt -s inherit_errexit` を有効にするとコマンド置換 `$(...)` 内でも `set -e` が効く（未設定だとサブシェル内エラーが無視される）
• `local var=$(command)` は `local` の戻り値（常に0）がコマンドのエラーを隠す罠。`local var; var=$(command)` と分けて書くのが安全
• `(( i++ ))` はiが0のとき評価結果が0でexitする。代わりに `i=$(( i + 1 ))` を使う

**git non-fast-forward push と rebase の操作順序**
• 自動化スクリプトでの鉄則は「変更を加える前に必ずpull」。`cp → pull → commit` の順はNGで、未コミット変更がある状態でpullするとrebaseがブロックされる。正しい順序は `pull → cp → commit → push`
• `git pull --rebase --autostash` を使うと、未コミット変更を自動でstashしてpull後にpop（restore）してくれる。自動化スクリプトではこの一行で安全に対応できる
• `git config --global rebase.autoStash true` をグローバル設定しておけば、以降の `git pull --rebase` が常にautostash付きで動作し、スクリプト側で意識しなくて済む
• 複数環境（手動・cron・CI）から同じリポジトリにpushする構成では、push直前に `git fetch origin && git status` で差分を確認するステップを入れるとトラブルを防げる
