**Claude Code サブエージェントへの MCP ツール伝播制限**
• サブエージェントはデフォルトで親会話の内部ツール・MCPツールを引き継ぐが、`tools` フィールド（許可リスト）を明示すると列挙されたツールのみに制限され、MCPツールはすべて除外される。例：`tools: Read, Grep, Glob, Bash` と定義したサブエージェントはMCPツールを一切使えない。
• `disallowedTools` を使えば「MCPツールは全部継承し、Write と Edit だけ禁止」という逆引きも可能。両方指定した場合は `disallowedTools` が先に適用され、残ったプールに対して `tools` が絞られる。
• lazyweb など特定MCPツールをサブエージェントに渡したい場合は、frontmatter の `mcpServers` キーで明示的に渡すか、`tools` フィールドを省略（= 全継承）にする必要がある。

**LIFF 審査中のローカル開発バイパスパターン**
• LINE 公式提供の `@line/liff-mock`（LIFF v2.19.0以降対応）を使うと、LIFF SDKをサーバーから切り離してモックデータで動作させられる。`liff.use(new LiffMockPlugin())` + `liff.init({ mock: true })` の2行で有効化でき、`liff.$mock.set()` で任意のプロフィールデータ等を注入できる。
• LINE Developer Console の審査中はエンドポイント変更不可という制約があるため、Cloudflare Tunnel でローカルを公開しても LIFF から確認できない。`NEXT_PUBLIC_USE_MOCK=true` のような環境変数でクライアント側のLIFF認証をスキップし、APIサーバー側も `mock-id-token-for-dev` トークンで分岐する実装パターンが最も現実的な回避策。
• 審査フェーズと開発フェーズを切り離すには「モックモードを本番コードに最初から仕込んでおく」が鉄則。環境変数で切替できる構造にしておくと、審査待ち期間中でも UI 開発をブロックされない。

**React/Next.js での外部画像 URL フォールバック設計（3段構え）**
• `<img>` タグの `onError` は「ネットワーク到達後に画像が壊れていた場合」にのみ発火する。`src` に `null` や `undefined` が渡った場合はブラウザが load error と判断せず `onError` が発火しないため、`src={url ?? fallbackUrl}` の null ガードを手前で必ず用意する必要がある。
• Unsplash の直接 ID 参照（`images.unsplash.com/photo-{ID}`）は削除・移動されたフォトIDで404になる。本番は Unsplash API から取得した `urls.regular` を使い、モック環境は確認済みIDを固定管理するのが安全。
• 堅牢な実装パターン：① `src` の null/undefined をフォールバック画像に差し替え → ② `onError` で別フォールバックに切替 → ③ `e.currentTarget.onError = null` でループ防止（無限再帰を防ぐ）の3段構えが推奨。
