**cronとPATH管理**
• cronは `/usr/bin:/bin` の最小限PATHで起動するため、`~/.zshrc` で設定したパス（Homebrew、アプリ独自binなど）は一切読み込まれない。crontab冒頭に `PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin` を明示するか、スクリプト内で絶対パスを使うのが鉄則。crontabで `PATH=$PATH:/追加パス` と書くと一部実装では `$PATH` がリテラル文字列扱いになりバグる。
• macOSではcronより**launchd**が推奨。plistの `EnvironmentVariables` キーでPATHを明示設定でき、さらに「スリープ中にスケジュール時刻を過ぎた場合、起動後に補完実行する」機能がある（cronはスキップするだけ）。
• 最も安全なパターンは「ラッパースクリプト」方式：cron/launchdからは薄いラッパーを呼び、そのラッパー冒頭でPATHをフルで定義してから本体を呼ぶ。ラッパー単体でも手動テスト可能で、環境差異が一箇所に集約される。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は「コマンド失敗で即終了」のつもりが、`if`/`elif`条件・`&&`/`||`リスト・パイプ非末尾コマンド・`!`反転などでは**無視される**。`local VAR=$(failing_cmd)` では `local` の終了コードが0を上書きするため、失敗が素通りする。
• Bash 4.4以降の `inherit_errexit` オプションと `-E`（errtrace）フラグを組み合わせると、サブシェル・関数・コマンド置換内でもerrexitとERRトラップを継承させられる。推奨セットは `set -euo pipefail` + `set -E` + `trap 'handle_error $LINENO' ERR`。
• `handle_error()` 関数で `${BASH_SOURCE[@]}` と `${BASH_LINENO[@]}` を出力するとスタックトレースがログに残る。`set -e` 単体への依存をやめ、ERRトラップを主軸にすることでデバッグ効率が大幅に上がる。

**git non-fast-forward と自動化スクリプトの操作順序**
• 自動化スクリプトでの正しい順序は `git stash → git pull --rebase → git stash pop → 作業 → git add → git commit → git push`。stash前にpullすると「uncommitted changesがある状態でrebaseがブロックされる」典型エラーが発生する。
• `git pull --rebase --autostash` フラグを使うとstash/popを自動化でき、操作ミスが減る。ただし stash pop でコンフリクトが起きた場合は終了コードを明示チェックすること。
• ソロ作業リポジトリでは `git config --global pull.rebase true` を設定しておくと、`git pull` だけでrebaseが走る。**push済みコミットのrebaseは禁止**（他ユーザーの履歴を書き換えるため）。
