**cronとPATH管理**
• cronジョブは `.zshrc`/`.bashrc` を読み込まない最小環境（`/usr/bin:/bin` のみ）で実行される。`PATH=$PATH:/custom/path` のような変数展開はcronでは機能しないため、crontabの先頭か各スクリプト内で `export PATH="/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin"` と絶対値で明示定義するのが鉄則。
• macOSのlaunchdも同様に最小環境で実行されるため、plistの `<key>EnvironmentVariables</key>` セクションでPATHを設定するか、ラッパーシェルスクリプト経由で起動するのがベストプラクティス。
• すべての実行コマンドに絶対パスを使う（例: `/opt/homebrew/bin/claude`）のが最も確実な回避策で、環境差異に依存しない。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` には重大な落とし穴がある。`local var=$(failing_cmd)` は `local` 自体の終了コードが0を返すためset -eが発動せずサイレントに失敗する。コマンド置換 `$()` 内でも Bash 4.4 以前は `inherit_errexit` 未設定だとset -eが継承されない。
• 推奨パターンは `set -eEu -o pipefail` + `trap 'echo "ERROR line $LINENO: exit $?" >&2' ERR`。`-E`（errtrace）でシェル関数・サブシェルにもERRトラップを継承、`-u` で未定義変数をエラー、`pipefail` でパイプ途中の失敗も検出できる。
• 意図的な非ゼロ終了は `some_cmd || true` や `if some_cmd; then ...; fi` で明示的に処理し、set -eのサイレント素通りを防ぐ設計が重要。

**git non-fast-forward と自動化スクリプトの操作順序**
• 自動化スクリプトでの正しいgit操作順序は「① ローカル変更をコミット（or `git stash`）→ ② `git pull --rebase` → ③ `git push`」。pullの前にunstaged changesがあるとrebaseがブロックされるため、必ずcommit or stashを先に行う。
• `git config --global pull.rebase true` でデフォルトをrebaseにするとmerge commitが発生せず履歴がすっきりする。ただしGit 2.34以降では `pull.ff only` との併用時に未コミット変更があるとabortするため、`git config pull.ff true` も追加設定が必要。
• 共有ブランチ（複数人が変更するブランチ）では `--rebase` は禁止。個人の自動化スクリプトやfeature branch限定で使用すること。
