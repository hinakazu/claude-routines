**cronとPATH管理**
• cronはユーザーの `.zshrc` / `.bash_profile` を読み込まず、デフォルトPATHは `/usr/bin:/bin:/usr/sbin:/sbin` のみ。Homebrew や `claude` コマンドなどはパスが通らずexit 127でサイレント失敗する
• 最も確実な対策はcrontabファイル冒頭に `PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin` を明示宣言するか、ラッパーシェルスクリプト内でPATHをセットしてから本体を呼ぶ方法。`PATH=$PATH:/追加パス` はcron実装によっては `$PATH` 展開が効かないため非推奨
• macOSではAppleがcronを非推奨とし、launchd推奨。launchdはplistファイルの `<key>EnvironmentVariables</key>` でジョブごとにPATHを明示宣言でき、スリープ復帰後の実行漏れ防止も含めて信頼性が高い

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体では if文条件式・`||` 演算子の左辺・`while` 条件内でコマンドが失敗してもトリガーされないため、予想外の箇所でスクリプトが静かに進行する。正しいパターンは冒頭に `set -euo pipefail` を置くこと（`-u`: 未定義変数エラー、`-o pipefail`: パイプ中間エラー検出）
• `trap 'echo "エラー: line $LINENO, exit $?" >&2' ERR` を追加すると失敗行番号と終了コードをstderrに出力でき、原因特定が大幅に速くなる
• `(( i++ ))` はカウンタが0になると終了コード1を返し `set -e` でスクリプトが終了する罠がある。`i=$(( i + 1 ))` または `(( i++ )) || true` に置き換えること
• 一時ファイルなどのクリーンアップは `trap 'rm -f "$TMPFILE"' EXIT` で宣言しておくと、正常終了・エラー終了・Ctrl+Cのすべてで確実に実行される

**git non-fast-forward push と rebase の操作順序**
• 「手元にコミット済み変更がある状態でpushしようとしたらリモートが先に進んでいた」ケースでは `git pull --rebase origin main` が正しい対処。mergeコミットを作らず線形ヒストリを保てる。`git config pull.rebase true` をグローバルに設定すると以降のpullが常にrebaseモードになる
• 自動化スクリプトでの安全な操作順序: ① `git stash push -m "作業内容"` → ② `git pull --rebase origin main` → ③ `git stash pop` → ④ ファイル操作 → ⑤ `git add / commit` → ⑥ `git push`。ファイルコピーをpullより先に行うとunstaged changesがrebaseをブロックしてハングする
• 2025年6月リリースの Gitea 1.25.0 では `ff_only` オプションが追加され、自動化パイプラインがnon-fast-forwardマージを誤って作成することを防ぐ設定が可能になった
