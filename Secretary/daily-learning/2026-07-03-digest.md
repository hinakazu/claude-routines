**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` / `.bashrc` を読み込まない最小環境（`/usr/bin:/bin` 程度）で実行されるため、HomebrewやNVM・rbenv等のツールが「手動では動くのにcronでは動かない」罠が発生する。対策は crontab 先頭に `PATH=/usr/local/bin:/usr/bin:/bin:/opt/homebrew/bin` を明示するか、スクリプト内でコマンドを絶対パスで指定すること。
• `#!/bin/bash -l`（ログインシェルオプション）をシェバン行に付けると `/etc/profile` や `~/.bash_profile` が読み込まれ、PATH問題を一括解消できる。
• macOSではcronはすでに非推奨。launchdへの移行が推奨される。launchdはMacがスリープ中にジョブがスキップされても、ウェイク後に自動実行する機能を持ち、エージェントごとに独立したEnvironmentVariables設定が可能なためPATH競合も起きない。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体には3つの盲点がある：①パイプライン中間コマンドの失敗（`cmd1 | cmd2` でcmd1が失敗しても検知しない）、②コマンド置換 `$()` 内のエラーは親シェルに伝播しない、③`((count--))` などの算術式で結果が0になるとスクリプトが終了する。
• 推奨ヘッダーは `set -euo pipefail`：`-u` は未定義変数の参照をエラーに、`-o pipefail` はパイプ内いずれかのコマンド失敗を検知する。3つ組で使って初めて「安全なスクリプト」になる。
• `trap 'handle_error $LINENO $?' ERR` でエラー発生時に行番号・終了コードをログに記録できる。`set -E`（errtrace）を追加すると関数・サブシェル内でもこのtrapが継承される。クリーンアップには `trap "cleanup" EXIT` を別途設定する。

**git non-fast-forward と自動化スクリプトの操作順序**
• `fatal: not possible to fast-forward` はローカルとリモートの両方に未共有のコミットが存在するとき（diverge状態）に発生する。自動化スクリプトで「ファイルコピー → commit → pull → push」の順にすると、pullした時点でunstagedな変更があり rebase がブロックされる。
• 正しい操作順序は `pull → ファイル操作 → add → commit → push`。pullを最初に行うことで最新のリモート状態からローカル変更を積み上げられる。
• `git pull --rebase` を使うとマージコミットが作られずに履歴がきれいに保たれる。unstaged changesがある場合は `git stash -u → pull → stash pop → commit → push` のパターンが安全。自動化スクリプトでは `git status --porcelain` で変更有無を事前確認する習慣をつけると競合事故を防げる。
