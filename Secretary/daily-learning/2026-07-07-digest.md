**cronとPATH管理**
• cronのデフォルトPATHは `/usr/bin:/bin` のみ。`.zshrc` は読み込まれないためHomebrewバイナリ（`/opt/homebrew/bin`等）がサイレントに失敗する。スクリプト冒頭で `PATH=/usr/local/bin:/opt/homebrew/bin:/usr/bin:/bin` のように完全パスを明示することが必須。`PATH=$PATH:/something` 形式は一部cronでbash変数展開が効かず危険。
• macOS 2025以降では **launchd** が公式推奨。`EnvironmentVariables` セクションでジョブごとにPATHを宣言でき、スリープ/ウェイク時のジョブ実行漏れもcronより確実に対応。イベントトリガー（ファイル変更等）にも対応しcronより柔軟。
• デバッグ時は `crontab -e` に `MAILTO=""` を追加し、スクリプト末尾に `>> /tmp/cron.log 2>&1` をつけてログを確認する。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` は &&/||リスト内・if文テスト部・while/until直後のコマンドでは**発動しない**というBash仕様上の穴がある。`local var=$(command)` も `local` が0を返すため失敗を隠蔽する典型的な罠。
• 推奨セットは `set -Eeuo pipefail`（-E=errtrace、-u=未定義変数エラー検知、-o pipefail=パイプ内エラー検知）。さらに `trap 'echo "ERROR at line $LINENO (exit $?)" >&2' ERR` を加えると失敗箇所をログに残せる。
• `-E`（errtrace）なしだと関数・コマンド置換内のERRトラップが継承されず、サブシェル内のエラーが無視される。Bash 4.4+ では `inherit_errexit` オプションで `$()` 内にも set -e を伝播できる。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forwardエラーは「自分がlast pullした後に別プロセス（CI自動コミット等）がリモートにpushした」場合に発生。自動化スクリプトでの正しい順序は **pull → ファイル変更 → commit → push**。uncommitted changesがある状態でpullするとrebaseがブロックされる。
• `git pull --rebase origin branch` → `git push origin branch` が最も整合した解消法。`git config --global pull.rebase true` を設定しておくと毎回 `--rebase` を指定する必要がなくなる。
• force pushが必要な場合は `--force` より `--force-with-lease` が安全。他者のコミットを上書きするリスクを防げる。rebase-firstワークフローを採用したチームはマージ競合が30〜40%減少するという報告もある。
