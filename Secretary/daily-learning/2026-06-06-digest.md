**Claude Code MCPツールとサブエージェントのスコープ制約**
• サブエージェントはデフォルトで親会話のツール権限を継承するが、frontmatterに `tools:` リストを明示すると **そのリストだけ** に制限される。lazyweb のようなプラグイン由来のMCPサーバーは、プラグインサブエージェントの場合セキュリティ上の理由で `mcpServers` / `hooks` / `permissionMode` フィールドが無視されるため、自動継承されない。
• 逆手に取ると、MCPサーバーをメイン会話のコンテキストに載せたくない場合はサブエージェントのfrontmatterに `mcpServers:` として定義することで、メイン会話を汚染せずサブエージェントだけに渡せる。
• サブエージェントはさらに別のサブエージェントを生成できない（無限ネスト防止）。つまりサブエージェントが使う特殊ツールは、生成する親が直接渡すかfrontmatterに記載する必要がある。

**LIFF審査中のローカル開発モックバイパスパターン**
• LINEが公式に `@line/liff-mock` パッケージを提供（LIFF SDK v2.19.0 以上）。`liff.use(new LiffMockPlugin())` 後に `liff.init({ liffId: '...', mock: true })` を呼ぶだけでLIFFサーバー依存をゼロにできる。
• `NEXT_PUBLIC_USE_MOCK=true` のような環境変数で `mock: process.env.NEXT_PUBLIC_USE_MOCK === 'true'` と制御するパターンが一般的。ビルド時に埋め込まれるため、Cloudflare Tunnel 等の公開URLなしでローカルでフル動作検証が可能。
• `liff.$mock.set()` でモックデータを動的に変更でき、`liff.$mock.clear()` でリセットできる。審査待ち期間中もUIロジックの実装・テストを止めずに進める体制が作れる。

**外部画像URL依存のフォールバック設計（React/Next.js）**
• Unsplash等の外部CDNのフォトIDは無効化・削除されることがある。`<img onError={handleError} />` パターンが基本だが、**エラーフラグを持たないと無限ループ**になる（フォールバック画像も失敗した場合、再度 onError が発火し続ける）。
• 実装パターン：`const [errored, setErrored] = useState(false)` を持ち、`onError={() => { if (!errored) { setErrored(true); setSrc(fallbackUrl); } }}` のように一度だけフォールバックに切り替える。`src` が `null`/`undefined` の場合は `onError` が発火しないため、レンダリング時に `!src` をチェックして即座にフォールバック表示する。
• 本番では「CDN → 自前サーバーの代替画像 → 汎用プレースホルダー」の多段フォールバックが推奨。フォールバック画像は `<link rel="preload">` でHTMLヘッドにプリロードしておくとレイアウトシフト（CLS）を防げる。
