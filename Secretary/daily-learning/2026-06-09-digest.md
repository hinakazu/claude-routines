**Claude Code MCPツールのサブエージェントスコープ制御**
• サブエージェントはデフォルトで親のMCPツールを継承するが、`tools`（許可リスト）または`disallowedTools`（拒否リスト）で制限できる。読み取り専用の調査エージェントには書き込みツールを渡さない設計が2026年のベストプラクティス。
• MCPサーバーをサブエージェントのfrontmatterに直接定義すると、親会話のコンテキストにそのツール説明が現れないためトークン節約になる。親からサブエージェント専用ツールを隔離する最も綺麗な方法。
• Skillはサブエージェントに自動継承されない。`skills`フィールドへの明示宣言が必要。また、サブエージェントが呼び出したSkill内でさらに`Agent()`を起動するとネストが折り畳まれるバグが報告されている（Issue #59968）。lazyweb ツールが桂小五郎に渡らなかった原因もこの制約に沿ったもの。

**LIFF審査中のローカル開発バイパスパターン**
• LINE公式の`@line/liff-mock`プラグイン（LIFF SDK v2.19.0以降）を使うと、`liff.init({ liffId: '...', mock: true })`でLIFFサーバーに依存しないモックモードが有効になる。`liff.$mock.set()`で任意のユーザープロフィール・シナリオを注入でき、審査待ち期間でも本番同等の動作確認が可能。
• LIFF CLIの`serve`コマンドはHTTPS付きローカルプロキシでエンドポイントURLを書き換えるが、公開済みLIFFアプリに実行すると一般ユーザーがアクセス不能になるため審査フロー中は使用禁止。`liff-mock` + `NEXT_PUBLIC_USE_MOCK=true`の組み合わせが安全な代替。
• `NEXT_PUBLIC_`変数はビルド時にJSバンドルへインライン展開されブラウザに露出する。モックフラグは`.env.local`（gitignore済み）で管理し、本番envには絶対に混入させないこと。

**React img onError フォールバック設計**
• 素朴な`onError={(e) => e.currentTarget.src = fallback}`は、フォールバック画像も壊れている場合に無限ループを起こす。`errored` stateフラグで二重発火を防ぐのが正しいパターン（SafeImageパターン）。
• `src` propsが変わった際に`useEffect`でerrorフラグをリセットしないと、別画像に遷移しても常にフォールバックが表示される。ナビゲーション対応が必須。
• `src`が`null`や`undefined`のときブラウザは`onError`を発火しないため、レンダリング前に`src == null`チェックをしてfallbackSrcをデフォルトとする処理が必要。CDN外部画像は「CDN → ローカルサーバー → 汎用プレースホルダー」の多段フォールバックが堅牢で、Unsplash IDの無効化にも対応できる。
