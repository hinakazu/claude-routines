**cronとPATH管理**
• cronジョブはユーザーの `.zshrc` を読み込まないため、`/usr/bin:/bin:/usr/sbin:/sbin` の最小PATHしか持たない。`claude` や `brew` 等のユーザーインストールコマンドは未登録になりexit 127で無音終了する。
• 解決策は2つ：①crontab冒頭に `PATH=/usr/local/bin:/usr/bin:/bin:...` と明示記述（変数展開は不可）、②ラッパースクリプト内でPATHを明示設定してから実際のジョブを呼び出す。
• macOSでは `launchd` が推奨。`~/Library/LaunchAgents/` の plist ファイルに `EnvironmentVariables` キーで環境変数を直接指定でき、スリープ復帰後もジョブをスキップしないcronより信頼性が高い。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` の3大落とし穴：①コマンド置換内（`$(...)`）では `-e` が継承されない（`shopt -s inherit_errexit` が必要）、②`local var=$(cmd)` で `local` の終了コードがマスクする、③パイプは最後のコマンドしか見ない（`-o pipefail` が必要）。
• 2026年のベストプラクティスは `set -euo pipefail` + `shopt -s inherit_errexit` + `trap 'echo "Error at line ${LINENO}, exit code $?" >&2' ERR` の組み合わせ。`set -E` を追加すると関数・サブシェル内でもERRトラップを継承できる。
• ShellCheck でスクリプトを静的解析するだけで上記の落とし穴の多くを事前検出できる。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーの根本原因：リモートが先に進んでいる状態でローカルにコミット済み変更があるとpushが拒否される。正しい操作順序は `git pull --rebase` → `ファイル操作` → `git add & commit` → `git push`。
• 自動化スクリプトでは `git pull --rebase` よりも `git pull --no-rebase`（merge strategy）が推奨される。rebaseはコミット履歴を書き換えるため、複数の自動化処理が同一リポジトリに書き込む場合に競合リスクが高い。
• 2026年のgit自動化ベストプラクティス：pre-push hookでテストとpull確認を自動実行、Conventional Commitsで機械可読なコミットメッセージ、Protectedブランチで誤った直接pushを防止。
