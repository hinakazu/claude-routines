**cronとPATH管理**
• cronはログインシェル（.zshrc等）を読み込まないため、デフォルトPATHは `/usr/bin:/bin` のみ。Homebrewやclaude等のカスタムコマンドは即座に「コマンドが見つからない（exit 127）」で無音終了する。
• 最も確実な対策はスクリプト冒頭でPATHを絶対パスで明示宣言すること：`PATH=/opt/homebrew/bin:/usr/local/bin:/usr/bin:/bin`（Apple Silicon Macは `/opt/homebrew/bin` 必須）。
• crontab自体の先頭行に `PATH=...` を書く方法もあるが、変数展開（`$HOME`等）は使えない制約がある。
• macOSでは Apple が10年以上前から launchd を推奨。スリープ後の実行漏れがなく、plistファイルの `EnvironmentVariables` キーでPATHを明示できる。ただしlaunchdは `~` を展開しないため、パスは必ず `/Users/yourname/` の形で絶対指定が必要。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` には「効かないケース」が複数ある：`if cmd; then`・`cmd && …`・`cmd || …`・`!cmd` のような条件式内のコマンドは失敗してもexitしない。また `(( i++ ))` は i=0 のとき評価値が0（false）のため意図せず即終了する。
• `|| true` パターン（例：`cd output || true`）はエラーを完全に隠蔽し、後続ステップのデバッグを困難にするアンチパターン。使うなら意図を明示してログに残す。
• ゴールドスタンダードは `set -euo pipefail` の3点セット：exit-on-error＋未定義変数エラー＋パイプライン途中の失敗検出。さらに `trap 'echo "Error on line $LINENO" >&2' ERR` を追加するとエラー行番号が即座にわかり、`bash -x script.sh` と組み合わせれば実行トレースも確認できる。
• `set -E` を加えると関数・サブシェル内でも ERR トラップが継承され、関数内部の失敗も捕捉できる。

**git non-fast-forward pushとrebaseの操作順序**
• non-fast-forward エラーは「ローカルが知らないリモートのコミットが存在する」状態で発生する。自動化スクリプトで複数箇所からpushする構成では特に頻発する。
• 正しい操作順序は **pull → ファイル操作 → commit → push** の一方向フロー。ファイルを変更してからpullすると、unstaged changesが残った状態でrebaseが始まり「Cannot rebase: You have unstaged changes」でブロックされる。
• 安全パターン：`git fetch origin` → `git rebase origin/main`（mainに追いつく）→ `git add/commit` → `git push`。もしくはpush前に `git stash -u` で変更を一時退避してからpullし、その後 `git stash pop`。
• force pushが必要になった場合は `--force-with-lease` を使う。通常の `--force` と違い、自分が知らない他者のコミットを上書きしない安全なオプション。CI/CDパイプラインではbranch同期チェックを組み込むことで未然に防止できる。
