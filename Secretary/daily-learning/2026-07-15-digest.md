**cronとPATH管理（macOS対応版）**
• cronジョブはユーザーの .bashrc/.zshrc を読み込まないため、PATH は /usr/bin:/bin のみ。Homebrew ツール（/opt/homebrew/bin）は使えず、`claude` 等が exit 127 でサイレント失敗する。crontab 冒頭に `SHELL=/bin/bash` と完全な PATH を明示的に書くか、スクリプト内で絶対パスを使うことで解消できる。
• macOS では Apple 公式推奨の launchd がより堅牢。plist の `EnvironmentVariables` キーで PATH を宣言でき、スリープ中に missed したジョブを起動後に自動リトライする機能も持つ。cron は missed jobs を再実行しないため、ラップトップ運用の定期自動化には launchd が優位。
• 防止策チェックリスト：①コマンドは絶対パスで記述、②スクリプト先頭で PATH を明示的に上書き、③crontab 内の `$PATH` 変数展開は使わない（実装依存で機能しないケースあり）。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` の主な落とし穴：①コマンド置換 `$()` 内では `set -e` が継承されない（Bash 4.4+ は `shopt -s inherit_errexit` で解決）、②`local var=$(cmd)` は `local` 自体が終了コード 0 を返すため失敗を隠蔽、③パイプの中間コマンドのエラーは無視される（`set -o pipefail` で解消）、④算術式 `(( i++ ))` が 0 評価のままスクリプト終了する。
• 推奨スクリプトヘッダは `set -euo pipefail`（-e: エラーで即終了、-u: 未定義変数をエラー、-o pipefail: パイプ失敗を伝播）＋ `trap 'echo "Error $? at line $LINENO: $BASH_COMMAND" >&2' ERR` の組み合わせ。失敗コマンド名・行番号・終了コードをすべて stderr に出力できる。
• クリーンアップには `trap cleanup EXIT` を使い、エラー終了・正常終了どちらでも一時ファイル削除等を確実に実行させる。診断出力は stderr、データ出力は stdout に分けることでログ収集とパイプ処理が両立する。

**git non-fast-forward push と rebase の操作順序**
• `git pull --rebase` は unstaged changes（ステージされていない変更）があるとブロックされる。自動化スクリプトで `cp → pull → commit → push` の順にすると、cp の結果が unstaged のまま pull を呼ぶため失敗する。正しい順序は `pull → 変更 → commit → push`。
• 自動化スクリプト向け 3 つの対策：①`git config --global rebase.autoStash true` でスタッシュ自動化（pull 前に自動 stash、pull 後に自動 pop）、②スクリプト冒頭で `git diff-index --quiet HEAD -- || { echo "unstaged changes"; exit 1; }` で事前チェック、③`git pull --rebase --autostash` フラグで 1 コマンド化。
• 複数箇所からの push が衝突する場合、`--force-with-lease` は upstream に想定外のコミットがあれば拒否する安全な force push。単純な `--force` より安全で、自動化スクリプトの標準的な選択。
