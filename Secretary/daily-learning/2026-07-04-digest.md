**cronとPATH管理**
• cronはユーザーの .zshrc/.bashrc を読まないため PATH が最小限（/usr/bin:/bin のみ）になる。Homebrew(/opt/homebrew/bin)や独自ツールが見つからずexit 127で無音終了するのが典型的な罠。解決策は①crontabの先頭でPATH=を明示、②スクリプト冒頭でPATH=を上書き、③絶対パスで呼び出す、の3択。
• macOSでは2005年以降 Apple が launchd を推奨。cron はMacがスリープ中に実行時刻を過ぎると「実行スキップ（ログなし）」だが、launchd はウェイクアップ時に遅延実行してくれる。
• launchd の plist ファイルには EnvironmentVariables ブロックでジョブごとにPATHを宣言でき、シェル設定に依存しない独立した実行環境を作れる。launchctl コマンドでジョブの状態・ログも確認できる点が cron より運用しやすい。

**`set -e` のサイレント終了とエラーハンドリング設計**
• set -e は非ゼロ終了で即終了するが例外が多く「安全に見えてデバッグを難しくする」落とし穴がある。①コマンド置換 $(cmd) はデフォルトで set -e を継承しない（Bash 4.4以降は inherit_errexit を追加して対処）、②local var=$(cmd) は local が成功扱いで終了コードをマスクする、③パイプライン(cmd1 | cmd2)は最後のコマンドの終了コードしか見ない。
• 正しい組み合わせは set -euo pipefail で、-u は未定義変数参照をエラーに、-o pipefail はパイプライン途中の失敗も検出する。
• エラーをログに残すには `trap 'echo "Error at line $LINENO: $BASH_COMMAND (exit $?)" >> "$LOG"' ERR` を冒頭に置く。$LINENO・$BASH_COMMAND・$? を組み合わせることで「どのコマンドが何行目でどのエラーコードで落ちたか」を記録できる。さらに set -x でコマンドを実行前に標準エラーに出力すると根本原因の特定が格段に速くなる。

**git non-fast-forward push と操作順序**
• 自動化スクリプトでの正しい git 操作順序は「pull → 変更作業 → commit → push」。pull を後回しにすると、リモートに新コミットがある状態で push しようとして non-fast-forward エラーになり、その後 pull --rebase しようとしても未コミット変更があってブロックされる二重の罠に陥る。
• コミット前に未コミット変更があるなら `git stash push → git pull --rebase → git stash pop → git commit → git push` の順序が安全。--force-with-lease は plain --force よりも安全で、リモートブランチが自分のローカル追跡と異なれば失敗してくれるため強制プッシュが必要な場面で推奨。
• `git config --global pull.rebase true` でデフォルトをリベース方式にしておくと、チーム全体でマージコミットを生成せずに線形履歴を維持できる。rebase を使っているチームはマージコンフリクトが30〜40%少ないという報告もある。
