**cronとPATH管理**
• cronジョブはユーザーの `.zshrc`/`.bash_profile` を読み込まないため、`/usr/local/bin` や `/opt/homebrew/bin` などのカスタムパスが失われ、コマンドが exit 127 で無音終了する。スクリプト冒頭で `PATH=/usr/local/bin:/usr/bin:/bin` を明示的に設定することが基本対策。
• macOSでは cron は非推奨で、Appleは **launchd** を公式に推奨している。launchd の `.plist` ファイルには `EnvironmentVariables` キーで環境変数を直接定義でき、スリープ/ウェイク時の実行欠落も防げる。
• 再利用性の高いパターンとして「ラッパースクリプト方式」がある：本体スクリプトを直接cronに登録せず、PATH設定・ログ出力・エラーキャッチを担うラッパーを介することで、テスタビリティと本番信頼性を両立できる。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体ではパイプライン・条件分岐・サブシェル内の失敗を検知しない。`set -euo pipefail` の三点セットに加え `shopt -s inherit_errexit` を追加することで、コマンド置換 `$(...)` 内の失敗も確実に伝播させられる。
• `trap 'handle_error $LINENO' ERR` でエラー発生時のハンドラを登録し、失敗した行番号・終了コードをログに記録するパターンが本番スクリプトの標準。`set -E`（errtrace）を併用すると関数内でも ERR トラップが継承される。
• 自動化スクリプトでは `set -e` を「安全機構」と誤信して終わりにするのが最大の落とし穴。クリティカルな処理は `cmd || { log_error "..."; exit 1; }` で個別にガードし、`set -e` はあくまで補助として使う設計が推奨される。

**git non-fast-forward と自動化スクリプトの操作順序**
• git 自動化スクリプトの正しい操作順序は必ず `fetch/pull → 作業（cp/編集）→ add → commit → push`。`pull` より先にファイルコピーや編集を行うと unstaged changes が rebase をブロックし、non-fast-forward エラーに至る。
• `git config --global pull.rebase true` でデフォルトの pull 戦略を rebase に変更することで、マージコミットを生まないクリーンな履歴を維持できる。force push が必要なケースでは `--force-with-lease` を使い、他者の push を上書きするリスクを排除する。
• 自動化スクリプトでは push が non-fast-forward で失敗した場合の**リトライロジック**を組み込むことが推奨される：push 失敗 → `git pull --rebase` → 再 push のループをリトライ上限付きで実装することで、複数箇所からの同時 push にも対応できる。
