**cronとPATH管理**
• cronはユーザーの .zshrc/.bashrc を読み込まず、`PATH=/usr/bin:/bin` のみの最小環境で起動する。`/usr/local/bin` や Homebrew 管理のバイナリは見つからず、exit 127 でサイレント失敗する。
• 対策は3段階：①crontab ファイル冒頭に `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` を追記、②スクリプト内でも明示 PATH 宣言、③バイナリはフルパスで呼び出す。
• デバッグ手順：`2>&1 >> /tmp/cron.log` でエラー出力をキャプチャ → `env -i HOME=$HOME bash --noprofile --norc -c "your-command"` で cron 同等環境を手動再現して確認。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` はゼロ以外の終了コードで即終了するが、`if` 条件式・`&&`/`||` 演算子・パイプ途中・`local var=$(失敗コマンド)` などの文脈では無視される。これが「手動では失敗するのにスクリプト内では握りつぶされる」現象の原因。
• 本番パターン：`set -euo pipefail`（未定義変数＋パイプ失敗検知）+ `set -E`（errtrace）+ `trap 'error_handler $LINENO' ERR` で失敗行・コマンド・終了コードをログに記録する。
• cron 向けには `2>> /var/log/myscript.log` でエラーをファイルに書き出し、`trap cleanup EXIT` で一時ファイルを必ずクリーンアップする構成が定番。

**git non-fast-forward push と rebase の操作順序**
• non-fast-forward エラーは「ローカルにない remote の commit が存在する」状態で push したときに発生。自動化スクリプトが「変更→commit→pull→push」の順で動くと、unstaged changes があるため rebase がブロックされる。
• 正しい順序は **pull → 変更 → commit → push**。自動化スクリプトでは `git pull --rebase` をファイル操作の前に実行するのが基本。
• 既にローカル変更がある状態で pull する場合は `git pull --rebase --autostash`（Git 2.6以降）で自動スタッシュ→rebase→復元できる。グローバル設定は `git config --global rebase.autoStash true`。rebase 後の force-push は必ず `--force-with-lease` を使い他人の push を上書きしないよう保護する。
