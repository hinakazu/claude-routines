**cronとPATH管理**
• cronはインタラクティブシェルではないため `.zshrc` / `.bashrc` が読み込まれず、カスタムPATHが未設定になる。デフォルトシェルは `/bin/sh`（dash）でありbashの機能も使えない。スクリプト冒頭で `SHELL=/bin/bash` を指定しない場合、bashism（配列・プロセス置換等）がサイレントに失敗する。
• 最も堅牢な対策は crontab のヘッダに `SHELL=/bin/bash` と `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` を明示的に記述すること。`PATH=$PATH:/custom/path` の形式は cronの実装によって変数展開されず `$PATH` がリテラル文字列になるため禁止。
• スクリプト内で完全パスを使う（`/usr/bin/python3` 等）か、シェバン行を `#!/bin/bash -l` にしてログインシェルを読み込む方法も有効。macOSの `launchd` はcronより環境変数の制御がしやすく、本番自動化ではlaunchdへの移行も選択肢。

**`set -e` のサイレント終了とエラーハンドリング設計**
• `set -e` 単体では `if` 条件式・`&&` チェーン・コマンド置換（`$()`）の中では発動しない。パイプラインでは最後のコマンド以外の失敗を無視する。これらを補うには必ず `set -euo pipefail` の3点セットで使うこと（`-u`=未定義変数エラー化、`-o pipefail`=パイプ途中の失敗を検知）。
• `trap ERR` でエラー発生時にハンドラを起動できるが、関数・サブシェル内では `set -E`（`errtrace`）を追加しないとトラップが伝播しない。実用的なパターン: `trap 'echo "Error on line $LINENO, exit: $?" >&2' ERR` をスクリプト冒頭に置き、ログ出力を保証する。
• 「失敗してもよい」コマンドには `cmd || true` を使い、意図的な非ゼロ終了と本当のエラーを明確に区別する。`set -e` を入れたまま cleanup 処理を確実に実行するには `trap cleanup EXIT` を組み合わせる。

**git non-fast-forward push と rebase の操作順序**
• "non-fast-forward" エラーはローカルブランチとリモートブランチが分岐した状態（両方に独自コミットがある）で発生する。`cp → commit → pull → push` の順序だと pull 時に unstaged changes があって rebase がブロックされる原因になる。正しい順序は **fetch/pull → ファイル変更 → add → commit → push**。
• 自動化スクリプトでは `git pull --rebase origin <branch>` を使うことでコミット履歴を線形に保てる。`git config --global pull.rebase true` を設定しておくと pull がデフォルトで rebase になる。また `git config rerere.enabled true` を有効にすると、過去に解決したコンフリクトのパターンを記憶して同じ衝突を自動解消できる。
• pre-push フックで「push前にリモートの最新コミットを fetch して差分チェック」する仕組みを入れると、push 失敗をゼロにできる。スクリプト例: `git fetch origin <branch> && git log HEAD..origin/<branch> --oneline | grep -q . && echo "Pull required before push" && exit 1`。
