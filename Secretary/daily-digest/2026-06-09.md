### Geminiのカテゴリ推論バイパス設計

**概要**: `resolveCategory()` がGeminiパスで `inferProductCategory` を実行し、ユーザーの `categoryHint`（乳液）を無視して商品名から推論した結果（クレンジング）を保存してしまうバグを修正。「AIに余計な推論をさせない」制約設計に転換した。

**なぜ深めたいか**: AIモデルを使ったデータ解析パイプラインで「いつAIに推論させ、いつ人間の入力を優先するか」の設計判断は汎用性が高い。過学習的な推論が引き起こすデータ汚染のパターンとして実務に直結する。

**推奨調査キーワード**: `Gemini API category override design pattern`, `AI pipeline human-in-the-loop constraints`, `LLM inference vs user input precedence`

### Prisma 7マイグレーション制約とチェックサム管理

**概要**: Prisma 7では適用済みマイグレーションに `migrate resolve` が使用不可のため、`_prisma_migrations` テーブルのチェックサムを直接 `SQL UPDATE` で更新するという制約回避手段を取った。

**なぜ深めたいか**: ORMのマイグレーション管理は本番DBで最もリスクの高い操作。「正規手段が使えないときの代替手段とそのリスク」を理解することは、スタートアップでDBAを兼任する場面で必須知識になる。

**推奨調査キーワード**: `Prisma migrate resolve alternative`, `Prisma 7 migration checksum manual update`, `Prisma migrations applied state repair`

### Gemini API 503/DEADLINE_EXCEEDEDのリトライ戦略

**概要**: Gemini APIが高負荷時に503を返し、商品解決が全件404になる問題に直面。DEADLINE_EXCEEDEDへのリトライを実装することで対応した。

**なぜ深めたいか**: 外部AIサービスの可用性は自社コントロール外で、サーキットブレーカーやカスケードフォールバック設計が製品の信頼性を決める。今回のcascade設計（Gemini失敗→次の手法へ）と組み合わせた堅牢化パターンとして体系化できる。

**推奨調査キーワード**: `Gemini API 503 retry strategy Next.js`, `exponential backoff external AI API`, `cascade fallback pattern product resolver`
