### Prisma マイグレーション管理の落とし穴

**概要**: Prisma 7 では適用済みマイグレーションに `migrate resolve` が使えず、`_prisma_migrations` テーブルのチェックサムを直接 SQL UPDATE するという代替手段が必要になった。セキュリティレビューで警告が検出された。

**なぜ深めたいか**: マイグレーション管理を誤ると本番DBの状態とスキーマ定義がズレる。正規の方法を把握しておかないと同じリスクを繰り返す。

**推奨調査キーワード**: `Prisma migrate resolve applied`, `Prisma 7 migration checksum`, `Prisma broken migration recovery`

### Gemini API エラーハンドリングとリトライ設計

**概要**: Gemini から 503 UNAVAILABLE・DEADLINE_EXCEEDED が返り、カスケード（DB→Gemini→URL→画像）が途中で止まる問題が発生。リトライ実装と Search Grounding の環境変数確認が必要だった。

**なぜ深めたいか**: 外部 LLM を本番で使う場合、一時的な高負荷やタイムアウトへの対処は必須設計。カスケード構造でどこでリトライ・フォールバックするかは汎用的な設計パターン。

**推奨調査キーワード**: `Gemini API 503 UNAVAILABLE retry strategy`, `exponential backoff LLM API`, `Gemini Search Grounding production setup`

---

### upsert のセマンティクスと update 句の罠

**概要**: Prisma の upsert で `create` にはカテゴリを含めていたが `update` 句には含めておらず、既存レコードのカテゴリが古い推論結果のまま残るバグが発生。`categoryOverride` を追加しても既存データには反映されなかった。

**なぜ深めたいか**: upsert は「新規なら insert、既存なら update」という単純な理解で使うと必ずこの罠にはまる。どのフィールドを update に含めるかを明示的に設計する習慣が必要。

**推奨調査キーワード**: `Prisma upsert update fields design`, `upsert partial update pitfall`, `idempotent data sync strategy`

### Supabase RLS ポリシーの設計と適用フロー

**概要**: `ProductRequest` テーブルに RLS ポリシー3本を追加したが、`db push` 済みのテーブルに対して migrate 経由で RLS のみ後から適用する必要があった。RLS のテスト・検証フローも課題として残った。

**なぜ深めたいか**: Supabase を本番で使う場合、RLS はセキュリティの最後の砦。マイグレーションと RLS 適用のタイミングがズレると穴が生まれる。正しい適用順序を体系化しておく価値が高い。

**推奨調査キーワード**: `Supabase RLS policy migration strategy`, `Row Level Security Supabase production`, `Prisma migrate RLS policy separate`
