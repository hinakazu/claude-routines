## 深掘りしたい学び

### cronとPATH管理
**概要**: cronジョブはユーザーのシェル環境を引き継がないため、`/Applications/cmux.app/Contents/Resources/bin` 等のカスタムPATHが参照できず `claude -p` が無音失敗していた。スクリプト冒頭でPATHを明示的に設定して解消した。
**なぜ深めたいか**: 自動化スクリプトが「動いているように見えて実は何もしていない」状態は気づきにくく、全自動配信系インフラの信頼性に直結する。
**推奨調査キーワード**: `cron PATH environment variable bash`, `cron job debugging macOS`, `launchd plist PATH setting`

### bashの `set -e` とエラーハンドリング設計
**概要**: `set -e`（エラー即終了）はサブシェル展開（`$(command)`）の失敗でも発動するため、エラーメッセージが標準エラーに流れたまま無音でスクリプトが死ぬ。`||`でフォールバックするか、`set +e`で局所的に無効化して対処した。
**なぜ深めたいか**: 自動化スクリプトで最も診断コストが高い障害パターン。ログが何も残らない失敗を設計で防ぐ知識は即実務転用できる。
**推奨調査キーワード**: `bash set -e subshell error handling`, `bash error handling best practices`, `bash trap ERR logging`

### non-fast-forward push と `git pull --rebase`
**概要**: 複数の書き込み元（ローカルcron・クラウドエージェント）が同じリポジトリに並行pushすると衝突する。`git pull --rebase` をpush前に挟むことでfast-forward状態を維持した。
**なぜ深めたいか**: 自動化の並列度が上がるほど頻発する。「pushが失敗する→次回も失敗」のサイレント負の連鎖を防ぐ設計知識として汎用性が高い。
**推奨調査キーワード**: `git pull rebase non-fast-forward automation`, `git push rejected rejected rejected fix`, `automated git workflow conflict prevention`
