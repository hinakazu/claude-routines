## 深掘りしたい学び

### cronとPATH管理
**概要**: cronジョブではユーザーの `.zshrc` が読み込まれないため、`/Applications/cmux.app/Contents/Resources/bin` 等が未登録になり、`claude` コマンドがexit 127で無音終了した。スクリプト冒頭で `PATH` を明示的に指定して解消。  
**なぜ深めたいか**: 自動化スクリプトが「手動では動くのにcronで動かない」という典型的な罠で、ローカル自動化を拡張するたびに踏む。事前に防ぐパターンを習得したい。  
**推奨調査キーワード**: `cron PATH environment variable best practices`, `bash script cron debugging`, `launchd vs cron macOS PATH`

### `set -e` のサイレント終了とエラーハンドリング設計
**概要**: `set -e` はコマンド失敗時に即終了するが、失敗ログを残さずに終了するため原因調査が困難だった。失敗コマンドの終了コードを明示的にキャッチしてログに書く方式に修正。  
**なぜ深めたいか**: `set -e` は安全に見えてデバッグを難しくする諸刃の剣。エラーハンドリングの設計パターンを体系化しておくと自動化スクリプトの信頼性が上がる。  
**推奨調査キーワード**: `bash set -e pitfalls`, `bash error handling trap ERR`, `bash script silent failure debugging`

### git non-fast-forward push と rebase の操作順序
**概要**: 複数箇所からpushが発生するリポジトリで、`cp → pull → commit → push` の順序ミスにより unstaged changes で rebase がブロックされた。正しい順序は `pull → cp → commit → push`。  
**なぜ深めたいか**: 自動化スクリプトでgit操作を組み合わせるとき、操作順序のミスが頻発する。git の状態機械を正しく理解することで同種のバグを未然に防げる。  
**推奨調査キーワード**: `git pull rebase before commit`, `git automation script best practices`, `git non-fast-forward error resolution`

### Claude CLIの非インタラクティブ実行（`claude -p`）
**概要**: `claude -p` をcronから呼び出すとハングする可能性があり、`--no-pager` や `--output-format` オプション、標準入力の扱いが重要になる。  
**なぜ深めたいか**: Claude CLIを自動化の部品として使う機会が増えており、非インタラクティブ環境での確実な動作パターンを押さえておくと応用範囲が広がる。  
**推奨調査キーワード**: `claude CLI non-interactive pipe usage`, `claude -p stdin automation`, `Claude Code headless execution`
