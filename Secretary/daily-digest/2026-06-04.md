### lazyweb によるデザインリサーチ自動化

**概要**: lazyweb MCP プラグインを使い、Votey・BuzzFeed・Kahoot などのスクリーンショットを取得して自分たちの UI 設計の裏取りに使った。サブエージェント（桂小五郎）には lazyweb ツールが渡らないため、メインの Claude が直接検索してから判断指示を渡す構成になった。

**なぜ深めたいか**: 「デザイン参考を探す → コンポーネント設計に落とす」をエージェントが自律的に行えると、デザイン決定のサイクルが大幅に短縮できる。ただし MCP ツールのスコープがサブエージェントに伝播しない制約を理解しておく必要がある。

**推奨調査キーワード**: `Claude Code MCP tool scoping subagent`, `lazyweb MCP design research workflow`, `Model Context Protocol tool inheritance agent`

### LIFF 審査中のローカル開発バイパスパターン

**概要**: LINE Developer Console でエンドポイント審査中はエンドポイント URL を変更できないため、Cloudflare Tunnel で localhost を公開しても LIFF から確認できなかった。`NEXT_PUBLIC_USE_MOCK=true` 環境変数でクライアント側認証をスキップし、API 側も `mock-id-token-for-dev` で分岐するパターンが既実装されていた。

**なぜ深めたいか**: LINE ミニアプリ開発では審査フローと開発サイクルが衝突しやすい。モックモードの設計パターンを正しく理解することで、審査待ち期間でも UI 開発を止めない体制を作れる。

**推奨調査キーワード**: `LIFF local development mock bypass`, `LINE Mini App審査中 localhost確認`, `Next.js環境変数 NEXT_PUBLIC mock development`

### Unsplash の画像 ID 管理とフォールバック設計

**概要**: Picsum の乱数写真から Unsplash のスキンケア特定フォト ID に切り替えたが、一部 ID が無効で画像が表示されなかった。`onError` ハンドラを img タグに追加し、壊れた ID は動作確認済み ID に差し替えた。

**なぜ深めたいか**: 外部画像 URL への依存はモック・本番問わず壊れやすい。CDN 依存の画像管理における堅牢性設計（onError・フォールバック・画像 CDN 選定）の原則を整理しておくと今後の開発で再発防止になる。

**推奨調査キーワード**: `Unsplash API photo ID stability`, `React img onError fallback pattern`, `外部画像URL依存 フォールバック設計`

### カード UI のアスペクト比設計原則

**概要**: チャレンジヒーローカードの画像がカード幅 358px に対して 120×120px の正方形で「切手」状態になっていた。lazyweb の参考事例と桂小五郎の判断を組み合わせ、full-width 16:9 横長に変更した。一覧カードのサムネイルは 64px 正方形が適切という結論も出た。

**なぜ深めたいか**: ヒーローカード vs 一覧カードという文脈でアスペクト比を使い分ける判断基準を言語化できると、次回の UI 設計で即適用できる。モバイルアプリにおける画像コンポーネントの設計ルールとして体系化する価値がある。
